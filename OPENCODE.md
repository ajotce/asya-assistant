# OPENCODE.md

> Этот файл — тонкая обёртка для opencode (https://opencode.ai). Все основные правила работы — в `AGENTS.md`. Здесь только opencode-специфика.

## Главное

opencode по умолчанию читает `AGENTS.md` в корне проекта — это и есть основной источник правил. Прочитай его перед началом работы.

После `AGENTS.md` прочитай `docs/roadmap.md` (раздел «Версия 1.0») и `docs/acceptance/v1.0.md` (создаётся в задаче 1.0.1).

Для пошаговых промтов мультиагентной разработки — `docs/v1.0-prompts.md`.

## opencode-специфика

### Использование разных моделей

opencode умеет работать с разными моделями (облачные провайдеры и локальные через Ollama). Для этого проекта рекомендуется:

- **Архитектурные задачи, миграции, security-критичный код, OAuth-флоу, wake-word** — самая сильная из доступных моделей.
- **Простые правки, форматирование, рутинные миграции, документация** — более лёгкая и дешёвая модель.
- **Никогда** не использовать локальные модели для задач с реальными секретами или критичной логикой авторизации/шифрования/cloud-deploy — слишком высок риск ошибок.

### Slash-команды и agents

Если используешь кастомные slash-команды или sub-agents в opencode — храни их конфиги в `.opencode/` и коммить вместе с проектом (но без секретов).

В фазе 1.0 полезные кастомные команды:
- `/cloud-audit` — быстрый аудит cloud-readiness (локальные пути, hardcoded конфиги).
- `/migration-check` — проверка свежих Alembic-миграций (один линейный chain, нет конфликтов).
- `/secrets-scan` — поиск возможных утечек секретов в коде/логах.

### Подтверждения перед изменениями

opencode может работать в режиме автономного выполнения. Для этого проекта **всегда** требуй подтверждения перед:

- Применением миграций (`alembic upgrade`/`downgrade`)
- Удалением файлов или записей в БД
- Изменением `.env`-файлов и секретов
- Push в удалённый репозиторий
- `terraform apply`, `terraform destroy` (фаза 1.0.5 — облачное развёртывание)
- `kubectl apply -f` против production-кластера
- Удалением S3-объектов и bucket'ов

### Параллельная работа

opencode может запускать несколько ассистентов параллельно (split-view / multi-pane). В фазе 1.0 это используется активно — см. `docs/v1.0-prompts.md`. Один агент = одна ветка `1.0/<task>-<name>`. Перед началом работы — `git pull origin 1.0-public`.

### Префикс коммитов

В фазе 1.0 — обязательно префикс задачи: `[1.0.3] PostgreSQL: ...`, `[1.0.9] OAuth Yandex: ...`.
