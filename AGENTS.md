# AGENTS.md

> Это канонический документ для AI-агентов (Claude Code, Codex, opencode, Cursor и любых других), работающих с проектом `ajotce/asya-assistant`. CLAUDE.md и OPENCODE.md — это тонкие обёртки, которые ссылаются на этот файл.

---

## Что такое Asya

Asya — персональный AI-ассистент. Сейчас проект находится в фазе **0.2: multi-user foundation**. Цель фазы — превратить одноюзерное локальное приложение в multi-user систему с БД, миграциями, авторизацией и шифрованием. Без открытой регистрации, без новых фич — только инфраструктура.

Полный план развития до 2.0 — в `docs/roadmap.md`. Прочитай его перед началом работы, чтобы понимать общий контекст.

---

## Текущая фаза и фокус

**Текущая фаза: 0.2 — Multi-user foundation.**

Что входит в фазу:

- БД (SQLite) с миграциями через Alembic.
- Multi-user модель: User, Session, Chat, Message с FK на user_id.
- Аутентификация: логин+пароль, без magic-link, без OAuth.
- Шифрование секретов в БД через master-key из `.env`.
- CLI для создания пользователей вручную (открытая регистрация откладывается на 0.4).
- Множественные чаты и Base-chat.
- Личный кабинет с базовыми настройками.
- Перенос текущего in-memory кода на БД.

Что **не** входит в фазу 0.2 (вообще не трогать):

- Память (профиль, эпизоды, правила, личность ассистента) — это 0.3.
- Голос — это 0.4 (push-to-talk) и 1.0 (wake-word).
- Дневник — это 0.4.
- Тихий наблюдатель — это 0.4.
- Telegram, Linear, Todoist, Gmail, GitHub, Bitrix24, Drive, Yandex.Disk, iCloud, OneDrive — это 0.4 и 0.5.
- Шаблоны документов — это 0.5.
- Magic-link / email-flow — исключён из проекта.

Если задача требует чего-то из «не входит» — **остановись и спроси пользователя**. Не реализуй ничего из этого без явного разрешения.

**Рабочая ветка:** `0.2-multi-user`. Все коммиты, относящиеся к 0.2, идут туда.

## Установка и запуск

Команды (могут уточняться при обновлении проекта; см. актуальные в `README.md`):

```bash
# Установка зависимостей backend
cd backend && pip install -r requirements.txt

# Установка зависимостей frontend
cd frontend && npm install

# Применение миграций БД
alembic upgrade head

# Создание пользователя через CLI (только в 0.2+, не в 0.1)
python -m backend.cli user create --login alice --password securepass

# Запуск через Docker Compose
docker compose up

# Запуск тестов
pytest backend/tests
cd frontend && npm test
```

Перед запуском убедись, что в `.env` заданы все обязательные переменные. Список — в `.env.example`.

## Workflow разработки

### Маленькие шаги

Работай **маленькими завершёнными шагами**. Не пытайся сделать всю задачу одним коммитом. Каждый шаг — это:

1. Понятное изменение в одном-двух файлах.
2. Запуск проверок (тесты, линтер, типы).
3. Если всё зелёное — коммит с осмысленным сообщением.

Если шаг получается большим (более 200 строк изменений) — разбей его дальше.

### Перед началом задачи — план

После изучения релевантных файлов — выложи в чат **короткий план** (3–7 пунктов): что собираешься сделать, в каком порядке, какие файлы затронешь. Жди подтверждения пользователя.

Если задача очевидно простая (например, поправить опечатку в документации) — план не нужен, делай сразу.

### Проверки после каждого шага

После каждого шага запускай:

- Backend: `pytest backend/tests`, `ruff check`, `mypy backend` (если настроен).
- Frontend: `npm test`, `npm run lint`, `npm run typecheck`.
- Миграции: при изменении моделей БД — `alembic check` и применение.

Если проверка не запускается (например, сервис недоступен) — **прямо скажи об этом в ответе**, не пропускай молча.

### Финальный ответ

В финальном ответе кратко напиши:

- **Что сделано** — 1–3 предложения.
- **Какие файлы изменены** — список путей.
- **Какие проверки выполнены** — какие команды и что вернули.
- **Что не проверено и почему** — честно.
- **Следующий шаг** — что делать дальше.

Структура — в свободной форме, но все 5 пунктов должны быть.

## Запреты

### Безопасность

- **Никогда не коммитить** в репозиторий: `.env`, API-ключи, токены, пароли, master-key, приватные файлы, временные данные.
- **Никогда не логировать** секреты — даже сокращённо, даже в DEBUG.
- **Никогда не отправлять** реальные секреты в LLM как часть запросов.
- В `.env.example` — только примеры без реальных значений.
- При работе с любым секретом — сразу шифровать через слой `backend/security/encryption.py`.

### Архитектура

- **Не делать** больших архитектурных изменений без явного обсуждения с пользователем. Большим считается: смена БД, переезд на новый фреймворк, реорганизация модулей затрагивающая >5 файлов.
- **Не добавлять** production-зависимости без необходимости. Если зависимость нужна — коротко объясни в комментарии к коммиту или в `docs/decisions.md`, зачем она.
- **Не использовать** локальные пути к файлам или абсолютные пути на конкретной машине. Всё через env-vars или конфиг.
- **Не хардкодить** ID, токены, имена ресурсов — только через конфиг или базу.

### Фичи вне фазы

- Не добавлять память, личность, голос, дневник, тихого наблюдателя, Telegram-бот, Linear, Todoist, Gmail, GitHub-интеграции, Bitrix24, документы — ничего, что относится к 0.3+.
- Если в задаче есть намёк на такую фичу — **остановись и уточни** у пользователя.

### Действия с данными

- **Никаких массовых удалений** в коде или скриптах без флага явного подтверждения.
- **Никаких** действий, изменяющих состояние сторонних сервисов (отправка email, создание PR, размещение постов) — в 0.2 этого вообще быть не должно.
- При сомнениях — выбирай самый безопасный вариант и объясни выбор.

## Стиль кода

### Backend (Python)

- Python 3.11+. Не использовать deprecated синтаксис.
- Type hints везде. `mypy --strict` — цель.
- Форматирование: `ruff format`. Линтер: `ruff check`.
- Импорты сортируются ruff-ом.
- Длина строки — 100 символов.
- Docstrings: Google-style для публичных функций и классов.
- Тесты: pytest, в `backend/tests/`. Один тест-файл на модуль кода.

### Frontend (TypeScript / React)

- TypeScript strict mode.
- Форматирование: prettier. Линтер: ESLint.
- React функциональные компоненты + хуки. Никаких classes.
- Стили: существующая конвенция проекта (см. код).

### Общее

- Имена переменных и функций — описательные. Не `data`, `info`, `result` без контекста.
- Комментарии на русском или английском — последовательно в одном файле. По умолчанию — английский для технических, русский для пользовательских строк.
- Никакого «волшебного» кода без комментария почему так сделано.

## Тесты

### Что тестировать обязательно

- Все слои аутентификации (логин, проверка прав, изоляция данных между пользователями).
- Все слои шифрования (encrypt/decrypt round-trip, обработка ошибок).
- Все CLI-команды.
- Миграции БД (upgrade и downgrade).
- Изоляцию пользователей (user A не видит данные user B по ID).

### Что покрытие важно поддерживать

- Сервисы (`SessionStore`, `ChatService`, и т.д.) — unit-тесты с in-memory SQLite.
- API endpoints — integration-тесты через TestClient.
- Frontend — компоненты с критической логикой.

### Что не нужно тестировать

- Тривиальные геттеры/сеттеры.
- Сторонние библиотеки.
- Логи и трейсинг.

## Документация

При изменении поведения проекта — обновляй документацию **в том же коммите** или сразу в следующем:

- Изменился API → обнови `docs/api.md`.
- Изменилась архитектура → обнови `docs/architecture.md`.
- Принято важное архитектурное решение → запиши в `docs/decisions.md` (формат ADR: контекст, решение, последствия).
- Изменился процесс разработки → обнови `docs/development.md`.
- Завершена крупная задача — добавь запись в `docs/development-log.md`.

Документация на русском языке (по предпочтению пользователя). Технические термины можно оставлять на английском, если у них нет устоявшегося русского аналога.

## Pull request / merge

- Каждая фаза (0.2, 0.3, ...) живёт в своей ветке (`0.2-multi-user`, `0.3-memory`, и т.д.).
- В пределах фазы — небольшие коммиты с осмысленными сообщениями.
- Перед merge ветки фазы в `main`:
  1. Все тесты зелёные.
  2. Все пункты `docs/acceptance/v0.X.md` пройдены.
  3. Документация обновлена.
  4. Нет TODO без issue в трекере.
  5. Нет закомментированного кода.
  6. CHANGELOG.md (если есть) обновлён.

## Уточнения и эскалация

### Когда задавать уточнения

- Без ответа продолжать **опасно** (можно случайно удалить данные, поломать продакшен, скомпрометировать секреты).
- Без ответа продолжать **невозможно** (не понятно что делать).
- Задача требует чего-то из «не входит в текущую фазу».
- Найден неожиданный архитектурный конфликт.

### Когда не задавать уточнения

- Можно выбрать самый безопасный минимальный вариант — выбирай и объясняй в ответе, какой вариант выбрал и почему.
- Это деталь стиля кода — следуй существующим конвенциям проекта.
- Это вопрос форматирования — следуй ruff/prettier.
