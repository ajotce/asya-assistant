# AGENTS.md — правила работы Codex над Asya Assistant

Этот файл задаёт обязательные правила для Codex / OpenAI Codex CLI и других coding agents при работе с репозиторием `ajotce/asya-assistant`.

Если указания пользователя, этот файл и текущие документы проекта конфликтуют между собой, агент должен остановиться, кратко описать конфликт и предложить безопасный вариант без потери данных и секретов.

## 1. Текущий фокус проекта

Проект переходит от локальной одноюзерной версии **Asya Local** к версии **Asya 0.2 — Multi-user foundation**.

Цель Asya 0.2: подготовить надёжный фундамент для мультипользовательской версии персонального ИИ-ассистента.
Все изменения в текущем этапе должны оцениваться через призму этой цели; scope 0.2 не расширяется без отдельного запроса пользователя.

В рамках Asya 0.2 нужно сосредоточиться на:
- SQLite + Alembic;
- SQLAlchemy-моделях и миграциях;
- пользователях, сессиях, сообщениях, файлах и usage-записях в БД;
- multi-user auth;
- регистрации по заявке / beta invite flow;
- magic-link и/или пароле;
- базовом личном кабинете пользователя;
- множественных чатах;
- обязательном базовом чате `Base-chat`;
- переносе текущих in-memory сущностей в БД;
- привязке пользовательских данных к `user_id`;
- шифровании секретов master-key из `.env`;
- сохранении текущего chat/file/retrieval/model-selection поведения.

В Asya 0.2 НЕ добавлять без отдельного явного запроса пользователя:
- долгосрочную память;
- формирование личности;
- новые внешние интеграции;
- голосовой режим;
- дневник;
- тихого наблюдателя;
- Telegram-бота;
- GitHub/Linear/Todoist/Gmail/Google Calendar интеграции;
- web search;
- мобильное приложение;
- сложную SaaS-биллинг/тарифную систему.

Важно: если новая задача выглядит полезной, но не укрепляет фундамент 0.2 (users/auth/chats/db/security), её нужно отложить до отдельного разрешения.

## 2. Обязательный старт каждой задачи

Перед изменениями агент должен:
1. Проверить, что открыт репозиторий `ajotce/asya-assistant`.
2. Проверить текущую ветку.
3. Для задач Asya 0.2 работать в ветке `0.2-multi-user`, если пользователь не указал другую ветку.
4. Прочитать `README.md`.
5. Прочитать `AGENTS.md`.
6. Прочитать `CLAUDE.md`, если он есть.
7. Прочитать актуальные документы в `docs/` по теме задачи.
8. Изучить текущую реализацию перед изменениями.
9. Кратко написать план на 3–7 пунктов.

Нельзя начинать переписывать архитектуру до изучения текущего кода.

## 3. Текущая техническая база

Сохранять и развивать существующую базу проекта:
- backend: `FastAPI`;
- frontend: `React + Vite + TypeScript`;
- локальный запуск: `Docker Compose`;
- same-origin раздача frontend через backend;
- VseLLM / OpenAI-compatible API для моделей;
- `VSELLM_API_KEY` хранится только на backend;
- текущие endpoint-ы `/api/health`, `/api/models`, `/api/settings`, `/api/session*`, `/api/chat/stream`, `/api/usage*` должны оставаться работоспособными или иметь документированную миграцию.

## 4. Архитектурные правила Asya 0.2

1. Делать изменения маленькими логическими шагами.
2. Не переписывать большие части проекта без необходимости.
3. Не смешивать бизнес-логику, storage-слой и UI.
4. Backend API должен быть явным, типизированным и документированным.
5. Новые БД-сущности проектировать так, чтобы позже можно было перейти с SQLite на PostgreSQL без полной переработки моделей.
6. Все пользовательские данные должны иметь связь с `user_id`, где это применимо.
7. Существующие session/file/vector/usage store-ы переносить в БД постепенно, сохраняя обратную совместимость на время миграции, если это снижает риск.
8. После добавления auth все пользовательские endpoints должны защищать данные от доступа другого пользователя.
9. `Base-chat` должен создаваться автоматически для каждого пользователя и не должен случайно удаляться при обычных операциях пользователя.
10. Для роли администратора заложить простую модель ролей (`user` / `admin`) без сложной RBAC-системы.

## 5. Зависимости

Не добавлять production-зависимости без причины.

Разрешённые для Asya 0.2 типы зависимостей, если они действительно нужны:
- SQLAlchemy;
- Alembic;
- библиотека для безопасного хеширования паролей;
- библиотека для подписи/шифрования токенов;
- библиотека для шифрования пользовательских секретов;
- FastAPI-Users или другой auth-пакет только после проверки совместимости с текущей архитектурой.

Если зависимость добавляется, агент должен кратко объяснить:
- зачем она нужна;
- почему нельзя обойтись текущими средствами;
- какие файлы обновлены;
- как проверить, что всё работает.

## 6. Auth и безопасность

1. Не хранить пароли в открытом виде.
2. Хранить только password hash.
3. Magic-link токены должны быть одноразовыми, ограниченными по времени и не должны логироваться целиком.
4. Сессионные cookies должны быть `HttpOnly`; для production предусмотреть `Secure`.
5. Не раскрывать `VSELLM_API_KEY`, master-key, session tokens, magic-link tokens в UI, логах, тестовых фикстурах и документации.
6. Master-key / encryption-key хранить только в `.env`; в `.env.example` указывать только пример имени переменной без реального значения.
7. Для пользовательских секретов использовать шифрование на backend.
8. Добавить проверки, что пользователь не может читать/изменять чужие чаты, файлы и usage-записи.

## 7. База данных и миграции

1. Все изменения схемы БД проводить через Alembic migration.
2. Миграции должны быть воспроизводимыми с чистого состояния.
3. Не удалять пользовательские данные миграциями без явного указания пользователя.
4. Для новых моделей добавить тесты на базовые операции создания/чтения/обновления/удаления.
5. Для перехода с in-memory на БД писать миграционный код аккуратно: сначала слой моделей и репозиториев, затем подключение endpoints.

Минимальные сущности Asya 0.2:
- `User`;
- `AuthSession` или аналог пользовательской web-сессии;
- `Chat` / `Session`;
- `Message`;
- `FileMeta`;
- `UsageRecord`;
- `BetaInvite` или `AccessRequest`, если реализуется заявка/инвайт;
- при необходимости `Secret` / `EncryptedSecret`.

## 8. Frontend-правила

1. Сохранять текущую простоту интерфейса.
2. Не превращать Asya 0.2 в большой редизайн.
3. Добавлять только UI, необходимый для:
   - входа;
   - регистрации / заявки на доступ;
   - выхода;
   - базового личного кабинета;
   - списка чатов;
   - создания/переименования/удаления чатов;
   - выбора `Base-chat` по умолчанию.
4. Текущие вкладки `Чат`, `Настройки`, `Состояние` должны продолжать работать.
5. Состояние чата не должно ломаться при переключении вкладок.

## 9. Документация

При изменении поведения обновлять релевантные документы:
- `README.md`;
- `docs/architecture.md`;
- `docs/api.md`;
- `docs/development.md`;
- `docs/testing.md`;
- `docs/development-log.md`;
- `docs/decisions.md`.

После каждого логически завершённого этапа добавлять запись в `docs/development-log.md`:
- дата;
- что изменено;
- какие проверки запущены;
- что не проверено и почему;
- риски/следующие шаги.

Не удалять старые MVP-документы без причины. Исторические материалы хранить в `docs/archive/`.

## 10. Проверки

Перед завершением задачи запускать доступные проверки:

```bash
make test
make lint
make build-frontend
```

Если нужны frontend unit-тесты:

```bash
docker run --rm -v "$PWD/frontend:/work" -w /work node:20-alpine sh -lc "npm ci && npm test"
```

Если проверка не запущена, честно указать причину в финальном отчёте и, когда уместно, в `docs/development-log.md`.

## 11. Git

1. Делать маленькие коммиты после логически завершённых шагов, если среда позволяет коммитить.
2. Использовать понятные сообщения коммитов.
3. Перед началом нового шага проверять `git status`.
4. Не коммитить `.env`, токены, временные файлы, логи, приватные данные и реальные ключи.
5. Не делать force-push без явного указания пользователя.
6. Перед merge в `main` убедиться, что проверки проходят и документация обновлена.

## 12. Как отвечать пользователю

1. Отвечать по-русски.
2. Писать простым языком, без лишнего жаргона.
3. Сначала кратко дать план, затем работать.
4. В финале сообщать:
   - что сделано;
   - какие файлы изменены;
   - какие проверки выполнены;
   - какие риски остались;
   - что делать следующим шагом.
5. Не скрывать технические проблемы.
6. Не обсуждать юридические вопросы, если пользователь специально не просит.

## 13. Главный принцип

Asya 0.2 — это не добавление новых возможностей ради возможностей. Это аккуратная подготовка фундамента: пользователи, БД, авторизация, чаты, безопасность и сохранение текущей функциональности.
