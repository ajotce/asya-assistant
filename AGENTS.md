# AGENTS.md — правила работы Codex над проектом Asya

Этот файл содержит обязательные инструкции для Codex / OpenAI Codex CLI при работе над проектом персонального ИИ-ассистента **Asya**.

Codex должен считать этот файл главным набором правил репозитория. Если какие-либо указания в текущей задаче конфликтуют с этим файлом, Codex обязан сообщить о конфликте пользователю и предложить безопасный вариант решения.

---

## 1. Главная цель проекта

Проект **Asya** — это MVP персонального ИИ-ассистента в формате PWA-веб-сервиса.

MVP должен быть реализован строго по документам:

1. `asya-mvp-tech-spec.md` — техническое задание MVP.
2. `asya-mvp-development-plan.md` — пошаговый план разработки.
3. `asya-codex-claude-start-prompt.md` — стартовый промт и рабочий порядок для Codex / Claude Code.

Главная цель MVP: создать рабочий текстовый чат-бот с выбором модели через VseLLM, временным контекстом сессии, поддержкой файлов и локальным backend в Docker.

---

## 2. Обязательный порядок работы

Перед началом любой разработки Codex обязан:

1. Прочитать `AGENTS.md`.
2. Прочитать `asya-mvp-tech-spec.md`.
3. Прочитать `asya-mvp-development-plan.md`.
4. Прочитать `asya-codex-claude-start-prompt.md`, если он есть в репозитории.
5. Кратко сформулировать, какой этап разработки выполняется сейчас.
6. Проверить, не противоречит ли задача пользователя ТЗ MVP.
7. Если задача выходит за рамки MVP, не начинать реализацию сразу, а явно указать это пользователю.

Codex должен работать пошагово, по этапам из плана разработки, а не пытаться реализовать весь проект одним большим изменением.

## 3. Что входит в MVP

В MVP входят:

- PWA-интерфейс для Safari на iPhone и Mac.
- Русский интерфейс.
- Ответы ассистента на русском и английском.
- Один пользователь.
- Текстовый чат.
- Потоковая генерация ответов.
- Глобальный выбор модели.
- Получение списка моделей из VseLLM API.
- Системный промт Asya, редактируемый в настройках.
- Временный контекст только в рамках backend-сессии.
- Загрузка файлов в текущую сессию.
- Поддержка PDF, DOCX, XLSX и изображений.
- До 10 файлов за одно сообщение.
- Размер каждого файла до 256 МБ.
- Embeddings + локальная временная векторная база для работы с файлами.
- Временное хранение файлов только в рамках сессии.
- Учёт токенов.
- Страница `Состояние Asya`.
- Минимальный health-check backend.
- Локальный запуск backend в Docker.
- Локальный режим, в котором backend может отдавать собранный frontend.
- Документация, поддерживаемая во время разработки.
- Автотесты для ключевой логики.

## 5. Рекомендуемый стек

Если в ТЗ не указано другое, использовать следующий стек:

- React.
- Vite.
- TypeScript.
- PWA.
- Responsive layout для iPhone и Mac.

- Без боковой панели чатов.

- Python.

- Нарезка документов на чанки.
- Создание embeddings.
- Локальный временный векторный индекс.
- Удаление файлов и индекса после завершения или очистки сессии.

## 6. Архитектурные правила

Codex обязан соблюдать следующие архитектурные правила:

1. Не хранить долговременную историю чатов.
2. Не создавать долговременную память пользователя.
3. Не сохранять факты о пользователе вне текущей сессии.
4. Не отправлять API-ключи во frontend.
5. Хранить VseLLM API key только на backend в `.env`.
6. Не коммитить `.env` и любые реальные секреты.
7. Всегда создавать и поддерживать `.env.example` без реальных ключей.
8. Разделять frontend и backend код.
9. Не смешивать бизнес-логику с UI-компонентами.
10. Делать backend API явным и документированным через OpenAPI.
11. Делать код расширяемым, чтобы в будущем можно было добавить Todoist, календарь, CRM, голос и память.
12. Не добавлять внешние облачные сервисы без явного разрешения пользователя.
13. Не добавлять веб-поиск в MVP.
14. Не менять выбранную архитектуру без причины и объяснения.

## 7. Локальный запуск и GitHub Pages

В MVP backend разворачивается только локально.

Из-за ограничений браузера и возможных проблем HTTPS GitHub Pages → локальный HTTP backend, Codex должен предусмотреть локальный режим, где backend может отдавать собранный frontend.

Правильная схема MVP:

1. Frontend разрабатывается как PWA.
2. Frontend может быть собран статически.
3. Backend в локальном режиме может отдавать собранный frontend.
4. GitHub Pages можно оставить как будущий режим публикации frontend, когда backend будет доступен по HTTPS.

Codex не должен делать GitHub Pages единственным способом запуска MVP.

## 8. Правила безопасности

Codex обязан:

1. Никогда не вставлять реальные API-ключи в код.
2. Никогда не коммитить `.env`.
3. Не выводить API-ключи в UI, логах или ошибках.
4. Маскировать секреты в логах.
5. Добавить `.gitignore` для `.env`, временных файлов, баз данных, кэшей, логов и артефактов сборки.
6. Хранить VseLLM API key только на backend.
7. Не добавлять авторизацию в MVP, так как backend локальный, но оставить архитектурную возможность добавить PIN/пароль позже.
8. Не открывать backend наружу без отдельной задачи по безопасности.
9. Логи могут содержать текст сообщений, но не должны содержать API-ключи и другие секреты.
10. Обрабатывать ошибки так, чтобы пользователь видел понятное сообщение, а разработчик — технические детали в логах.

---

## 9. Правила работы с VseLLM

Codex должен реализовывать VseLLM как OpenAI-compatible API.

Основные настройки:

- `VSELLM_API_KEY` — ключ API.
- `VSELLM_BASE_URL` — базовый URL API.
- Значение по умолчанию для base URL: `https://api.vsellm.ru/v1`.

Обязательные возможности:

1. Получение списка моделей через API.
2. Глобальный выбор модели в настройках.
3. Chat completions для обычного текстового чата.
4. Streaming ответов.
5. Embeddings для файлового поиска.
6. Учёт токенов, если API возвращает usage.
7. Если API не возвращает цену, описание или контекст модели, показывать только ID модели.

Codex не должен хардкодить конкретные модели как единственный доступный список.

---

## 10. Правила frontend-разработки

Frontend должен быть простым, устойчивым и удобным для пользователя, который не является программистом.

Обязательные экраны:

1. Главный экран чата.
2. Настройки.
3. Страница `Состояние Asya`.

Главный экран чата должен поддерживать:

- поле ввода сообщения;
- отправку сообщения;
- потоковое отображение ответа;
- редактирование последнего пользовательского сообщения и повторную генерацию;
- загрузку файлов;
- отображение прикреплённых файлов;
- понятные ошибки;
- кнопку очистки текущей сессии.

Настройки должны поддерживать:

- имя ассистента;
- выбор модели;
- просмотр списка доступных моделей;
- редактирование системного промта;
- отображение статуса VseLLM API key без раскрытия ключа.

Страница `Состояние Asya` должна показывать:

- backend online/offline;
- выбранную модель;
- статус VseLLM API key;
- статус получения моделей;
- статус embeddings;
- статус временного хранилища;
- версию приложения;
- uptime backend, если доступен.

---

## 11. Правила backend-разработки

Backend должен предоставлять понятный API.

Минимальные группы endpoint'ов:

- `/health` — проверка состояния backend.
- `/models` — список моделей VseLLM.
- `/settings` — настройки Asya.
- `/chat` — отправка сообщений и streaming ответов.
- `/files` — загрузка и обработка файлов.
- `/session` — управление временной сессией.
- `/usage` — токены и статистика использования.

Backend должен:

1. Валидировать входные данные.
2. Ограничивать размер и количество файлов.
3. Возвращать понятные ошибки.
4. Поддерживать OpenAPI/Swagger.
5. Иметь health-check.
6. Иметь модульную структуру.
7. Разделять API routes, services, settings, models/schemas, storage и integrations.
8. Поддерживать local/prod конфигурации, даже если production пока не используется.

---

## 12. Правила работы с файлами

Файлы в MVP:

- PDF;
- DOCX;
- XLSX;
- изображения.

Ограничения:

- максимум 10 файлов за одно сообщение;
- максимум 256 МБ на один файл;
- файлы хранятся только в рамках временной сессии.

Codex обязан:

1. Проверять тип файла.
2. Проверять размер файла.
3. Извлекать текст из поддерживаемых документов.
4. Для XLSX учитывать листы и таблицы.
5. Нарезать документы на чанки.
6. Создавать embeddings.
7. Хранить временный индекс только в рамках сессии.
8. Удалять временные файлы при очистке сессии.
9. Понятно сообщать пользователю, если файл не удалось обработать.

Если изображение отправляется в модель, Codex должен проверять, поддерживает ли выбранная модель vision-вход. Если не поддерживает, пользователь должен получить понятное сообщение.

## 13. Правила поведения Asya

Поведение Asya в MVP:

- деловое;
- дружелюбное;
- без проактивных уведомлений;
- без самостоятельных действий вне ответа на сообщение пользователя.

Asya должна использовать гибридную стратегию уточнений:

1. Если задачу можно выполнить без уточнений — выполнить best effort.
2. Если есть неоднозначность, но можно сделать разумное допущение — сделать допущение и явно его указать.
3. Если без уточнения невозможно выполнить задачу — задать вопрос.
4. Если задача связана с риском потери данных, ошибками в документах, деньгами или безопасностью — уточнить до выполнения.

Asya не должна сама создавать напоминания, инициировать действия или предлагать внешние интеграции в MVP.

## 14. Тестирование

Codex обязан добавлять и поддерживать тесты для ключевой логики.

Минимальные тесты backend:

- health-check;
- загрузка настроек;
- получение моделей;
- обработка ошибок VseLLM;
- создание сессии;
- очистка сессии;
- валидация файлов;
- ограничение размера файлов;
- ограничение количества файлов;
- извлечение текста из файлов;
- базовый embeddings pipeline;
- streaming chat API, если тестируемо.

Минимальные тесты frontend:

- рендер главного экрана;
- отправка сообщения;
- отображение streaming-состояния;
- настройки модели;
- редактирование системного промта;
- страница состояния;
- отображение ошибок.

Перед завершением этапа Codex обязан запускать доступные тесты и сообщать результат.

Если тесты невозможно запустить, Codex обязан явно написать причину.

## 15. Документация

Codex обязан поддерживать документацию во время разработки.

Минимальные файлы документации:

- `README.md` — как запустить проект локально.
- `.env.example` — пример переменных окружения.
- `docs/architecture.md` — архитектура проекта.
- `docs/development-log.md` — журнал разработки.
- `docs/api.md` — описание API, если OpenAPI недостаточно.

Запись в development log должна включать:

- дату;
- что сделано;
- какие файлы изменены;
- какие тесты запущены;
- какие проблемы остались;
- следующий рекомендуемый шаг.

## 16. Git и изменения

Codex должен:

1. Делать изменения небольшими логическими шагами.
2. Не переписывать большие части проекта без необходимости.
3. Не удалять существующий код без объяснения.
4. Не менять публичные API без обновления документации и тестов.
5. Перед крупным изменением кратко объяснять план.
6. После изменения кратко объяснять, что было сделано и как проверить.

Если используется Git, рекомендуемые сообщения коммитов:

- `chore: initialize project structure`
- `feat: add backend health check`
- `feat: add VseLLM model loading`
- `feat: add streaming chat endpoint`
- `feat: add frontend chat screen`
- `feat: add file upload pipeline`
- `test: add backend API tests`
- `docs: update development log`

Codex не должен создавать бессмысленные коммиты вроде `update files` или `fix stuff`.

## 17. Критерии готовности этапа

Этап считается готовым только если:

1. Код реализован.
2. Код соответствует ТЗ.
3. Нет известных критических ошибок.
4. Запущены доступные тесты.
5. Результаты тестов сообщены пользователю.
6. Обновлена документация, если поведение проекта изменилось.
7. Не добавлены реальные секреты.
8. Не нарушены ограничения MVP.

Если хотя бы один пункт не выполнен, Codex должен явно сообщить, что этап не завершён полностью.

---

## 18. Критерии готовности всего MVP

MVP считается готовым, когда:

1. Проект запускается локально одной понятной командой или набором команд из README.
2. Backend запускается в Docker.
3. Frontend открывается локально через backend или корректно подключается к backend в local-режиме.
4. Пользователь может открыть Asya на Mac.
5. Пользователь может открыть Asya на iPhone в локальной сети, если это технически настроено.
6. Можно выбрать модель VseLLM.
7. Можно отправить сообщение.
8. Ответ приходит потоково.
9. Можно изменить системный промт.
10. Можно загрузить поддерживаемые файлы.
11. Asya может отвечать с учётом загруженных файлов.
12. Контекст работает только в рамках текущей backend-сессии.
13. Можно очистить текущую сессию.
14. Страница `Состояние Asya` показывает реальные статусы.
15. `/health` работает.
16. Реальные API-ключи не попали в репозиторий.
17. Документация актуальна.
18. Базовые тесты проходят или известные проблемы явно задокументированы.

## 19. Как отвечать пользователю

Codex должен отвечать пользователю простым языком на русском.

Пользователь не пишет код самостоятельно, поэтому Codex должен:

- объяснять, что было сделано;
- говорить, какие команды нужно выполнить;
- не перегружать ответ лишними техническими деталями;
- явно писать, если требуется ручное действие пользователя;
- не просить пользователя принимать технические решения, если можно выбрать разумный вариант на основе ТЗ;
- если есть несколько вариантов, давать рекомендацию.

## 21. Если задача неясна

Если задача пользователя неясна, Codex должен:

1. Проверить ТЗ и план разработки.
2. Сделать разумное предположение, если риск низкий.
3. Явно написать принятое предположение.
4. Задать уточняющий вопрос только если без ответа невозможно продолжить или есть риск серьёзной ошибки.

По умолчанию Codex должен двигать разработку вперёд, а не останавливать работу из-за мелких неопределённостей.

## 22. Главный принцип

Главный принцип работы Codex над Asya:

> Сначала сделать стабильный, простой и понятный MVP, строго соответствующий ТЗ, без лишних функций и без усложнения архитектуры.
