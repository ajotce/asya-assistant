# AGENTS.md — правила работы Codex над Asya Local

Этот файл содержит обязательные инструкции для Codex / OpenAI Codex CLI при работе над проектом **Asya Local**.

Если указания в текущей задаче конфликтуют с этим файлом, нужно явно сообщить о конфликте пользователю и предложить безопасный вариант.

## 1. Назначение проекта

**Asya Local** — локальная версия персонального ИИ-ассистента Asya в формате PWA с backend.

## 2. Порядок работы

Перед началом изменений Codex должен:
1. Прочитать `AGENTS.md`.
2. Прочитать `README.md`.
3. Прочитать актуальные документы в `docs/`.
4. Кратко определить, какой участок системы изменяется.

Во время работы:
1. Делать небольшие логические изменения.
2. Сначала изучать контекст, затем вносить правки.
3. Запускать доступные проверки и тесты.
4. Явно сообщать, если часть проверок не была запущена.
5. Обновлять документацию, если поведение системы изменилось.

## 3. Базовые технические правила

1. Не менять архитектуру без понятной причины и объяснения.
2. Держать frontend и backend разделёнными.
3. Не смешивать бизнес-логику и UI.
4. Backend API поддерживать явным и документированным.
5. Сохранять локальный запуск через Docker как основной рабочий режим.
6. Не переписывать большие части кода без необходимости.

## 4. Безопасность

1. Никогда не добавлять реальные API-ключи в репозиторий.
2. Никогда не коммитить `.env`.
3. Хранить `VSELLM_API_KEY` только на backend.
4. Не раскрывать API-ключи в UI, логах и ответах пользователю.
5. Поддерживать `.env.example` без реальных значений.

## 5. Документация и тесты

1. Поддерживать в актуальном состоянии:
   - `README.md`
   - `docs/architecture.md`
   - `docs/api.md`
   - `docs/development.md`
   - `docs/testing.md`
   - `docs/development-log.md`
   - `docs/decisions.md`
2. После логически завершенного этапа добавлять запись в `docs/development-log.md`.
3. Перед завершением задачи запускать доступные проверки (`make test`, `make build-frontend`, `make lint`, smoke при необходимости).
4. Если проверка недоступна, фиксировать причину в отчёте и в `docs/development-log.md`.

## 6. Git и изменения

1. Изменения должны быть маленькими и завершёнными.
2. Не удалять существующий код или документы без причины.
3. Не менять публичные API без обновления документации и тестов.
4. Использовать понятные сообщения коммитов.

## 7. Как отвечать пользователю

1. Отвечать простым русским языком.
2. Кратко объяснять, что сделано.
3. Сообщать, какие проверки запущены и что не удалось проверить.
4. Не скрывать технические риски и ограничения.

## 8. Главный принцип

Сначала стабильность и понятность текущей версии Asya Local, затем развитие без необоснованного усложнения.
