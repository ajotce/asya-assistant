# AGENTS.md — правила работы Codex над Asya Assistant

Этот файл задаёт обязательные правила для Codex / OpenAI Codex CLI и других coding agents при работе с репозиторием `ajotce/asya-assistant`.

Если указания пользователя, этот файл и текущие документы проекта конфликтуют между собой, агент должен остановиться, кратко описать конфликт и предложить безопасный вариант без потери данных и секретов.

## 1. Текущий фокус проекта

Проект переходит от **Asya 0.2 — Multi-user foundation** к **Asya 0.3 — Memory and Personality**.

Базовая ветка текущего этапа: `0.2-multi-user`.

Цель Asya 0.3: добавить управляемую долговременную память, базовую систему личности ассистента, пользовательские пространства, ленту памяти и прозрачную ленту активности без внешних интеграций, голосового режима и дневника.

В рамках Asya 0.3 нужно сосредоточиться на:
- профиле пользователя: структурированные факты со статусами доверия;
- эпизодической памяти: важные фрагменты диалогов с embeddings;
- правилах поведения: предпочтения пользователя и правила ассистента;
- базовом профиле личности ассистента;
- адаптации тона ответа с учётом памяти и пространства;
- ленте памяти с возможностью просмотра, подтверждения, редактирования, запрета использования и отката;
- версионировании изменений памяти;
- снэпшотах памяти;
- пользовательских пространствах;
- per-space настройках памяти и личности;
- admin-only пространстве `Asya-dev`;
- ленте активности ассистента;
- использовании релевантной памяти/правил/личности в chat context;
- сохранении стабильности auth/chats/files/retrieval/usage из версии 0.2.

В Asya 0.3 НЕ добавлять без отдельного явного запроса пользователя:
- внешние интеграции: Google, Apple, Gmail, Todoist, Linear, Bitrix24, GitHub, Telegram;
- голосовой режим, wake word, STT/TTS;
- дневник и голосовой дневник;
- тихого наблюдателя;
- систему обещаний и обязательств;
- центр уведомлений;
- конструктор сценариев;
- шаблоны документов;
- приватный чат с E2E/zero-knowledge шифрованием;
- SaaS-биллинг и тарифы;
- web search;
- мобильное приложение.

Важно: v0.3 должна сделать Asya “помнящей”, но не должна превращаться в интеграционный комбайн раньше времени.

## 2. Обязательный старт каждой задачи

Перед изменениями агент должен:
1. Проверить, что открыт репозиторий `ajotce/asya-assistant`.
2. Проверить текущую ветку.
3. Для задач Asya 0.3 работать от ветки `0.2-multi-user`, если пользователь не указал другую ветку.
4. Проверить `git status`.
5. Прочитать `README.md`.
6. Прочитать `AGENTS.md`.
7. Прочитать `CLAUDE.md`, если он есть.
8. Прочитать релевантные документы в `docs/`.
9. Изучить текущую реализацию кода, который будет изменяться.
10. Кратко написать план на 3–7 пунктов.

Нельзя начинать переписывать архитектуру до изучения текущего кода.

## 3. Техническая база

Сохранять и развивать существующую базу проекта:
- backend: `FastAPI`;
- frontend: `React + Vite + TypeScript`;
- локальный запуск: `Docker Compose`;
- same-origin раздача frontend через backend;
- модели и embeddings через VseLLM / OpenAI-compatible API;
- `VSELLM_API_KEY` хранится только на backend;
- БД: SQLite + SQLAlchemy + Alembic;
- multi-user auth и изоляция данных по `user_id`;
- множественные чаты и обязательный `Base-chat`;
- user-scoped доступ к chats/messages/files/usage.

Текущие endpoint-ы `/api/health`, `/api/models`, `/api/settings`, `/api/session*`, `/api/chat/stream`, `/api/chats*`, `/api/usage*`, `/api/auth*` должны оставаться работоспособными или иметь документированную миграцию.

## 4. Архитектурные правила Asya 0.3

1. Делать изменения маленькими логическими шагами.
2. Не переписывать большие части проекта без необходимости.
3. Не смешивать business logic, storage/repository слой и UI.
4. Backend API должен быть явным, типизированным и документированным.
5. Все пользовательские данные памяти должны быть связаны с `user_id`.
6. Все данные пространства должны быть связаны с `space_id`, если сущность относится к пространству.
7. Не допускать утечек памяти между пользователями.
8. Не допускать использования памяти из приватных/запрещённых источников.
9. Память должна быть управляемой: пользователь видит, что записано, и может запретить использование.
10. Любое изменение памяти должно быть журналируемым и откатываемым.
11. Retrieval памяти должен быть ограничен текущим пользователем и, при необходимости, текущим пространством.
12. Не добавлять фоновые очереди/Celery без необходимости. Если нужен автоматический процесс, сначала предпочесть простой synchronous/lazy механизм.
13. Новые зависимости добавлять только если без них нельзя надёжно реализовать задачу.

## 5. Модель памяти v0.3

Минимальные домены памяти:
- `UserProfileFact`: факт о пользователе.
- `MemoryEpisode`: эпизод из диалога или важный контекст.
- `MemoryChunk` / embedding-запись для поиска по памяти.
- `BehaviorRule`: правило поведения ассистента.
- `AssistantPersonalityProfile`: базовый профиль личности ассистента пользователя.
- `SpaceMemorySettings`: per-space настройки использования и записи памяти.
- `MemoryVersion` / `MemoryChange`: история изменений.
- `MemorySnapshot`: снэпшоты состояния памяти.
- `ActivityLog`: прозрачная лента действий ассистента.

Статусы памяти:
- `confirmed` — явно подтверждено пользователем;
- `inferred` — выведено ассистентом из диалога;
- `needs_review` — требует подтверждения;
- `outdated` — устарело;
- `forbidden` — запрещено использовать;
- `deleted` — скрыто/удалено логически, если физическое удаление не реализовано.

Правило по умолчанию:
- если пользователь явно говорит “запомни” — можно сохранять как `confirmed`;
- если ассистент сам извлекает факт — сохранять как `inferred` или `needs_review`;
- если пользователь говорит “забудь” — переводить в `forbidden` или `deleted` согласно текущей реализации;
- память не должна использоваться в ответах, если она `forbidden`, `deleted` или не принадлежит текущему пользователю.

## 6. Личность ассистента v0.3

v0.3 реализует только базовую управляемую личность, без сложной “эволюции живого персонажа”.

Разрешено:
- адаптировать тон под предпочтения пользователя;
- учитывать имя ассистента и стиль пользователя;
- использовать лёгкий юмор, если это уместно;
- мягко возражать пользователю, если есть риск ошибки;
- извиняться при ошибке и предлагать сохранить правило, чтобы не повторять её;
- иметь базовый профиль личности + overlay пространства.

Не делать:
- не имитировать сознание, эмоции или “живую сущность”;
- не играть роль терапевта;
- не лезть в психологическое состояние пользователя без явного запроса;
- не делать сложную автономную эволюцию личности.

## 7. Пространства v0.3

В v0.3 нужно добавить пользовательские пространства как отдельный контекст работы.

Минимально:
- у пользователя есть базовое пространство по умолчанию;
- пользователь может создавать, переименовывать и архивировать свои пространства;
- чаты могут быть связаны с пространством;
- у пространства есть настройки памяти: читать память, записывать память, использовать правила, использовать personality overlay;
- `Asya-dev` — служебное admin-only пространство только для главного администратора;
- обычный пользователь не видит и не может создать/удалить `Asya-dev`.

Не добавлять в v0.3:
- общие пространства между несколькими пользователями;
- сложные роли внутри пространства;
- файловые источники пространства;
- внешние интеграции пространства.

## 8. Безопасность

1. Никогда не коммитить `.env`, API-ключи, токены, приватные данные и временные файлы.
2. Не выводить секреты в логах и пользовательских ответах.
3. Использовать `.env.example` только с пустыми или демонстрационными значениями.
4. Все memory/activity/space endpoint-ы должны проверять владельца через `user_id`.
5. Admin-only функциональность должна быть защищена ролью пользователя.
6. Не хранить reasoning/thinking моделей как обычное сообщение или память, если это уже запрещено текущей архитектурой.
7. Не записывать чувствительные данные в memory автоматически, если они похожи на пароли, токены, ключи, приватные документы, платёжные данные.
8. Не допускать cross-user data leakage в retrieval, memory feed и activity feed.

## 10. Проверки

Перед завершением задачи запускать доступные проверки:

```bash
make test
make lint
make build-frontend
```

Если нужны frontend unit-тесты:

```bash
docker run --rm -v "$PWD/frontend:/work" -w /work node:20-alpine sh -lc "npm ci && npm test"
```

Если проверка не запущена, честно указать причину.

## 11. Git

1. Делать маленькие коммиты после логически завершённых шагов, если среда позволяет коммитить.
2. Использовать понятные сообщения коммитов.
3. Перед началом нового шага проверять `git status`.
4. Не коммитить `.env`, токены, временные файлы, логи, приватные данные и реальные ключи.
5. Не делать force-push без явного указания пользователя.
6. Перед merge в `main` убедиться, что проверки проходят и документация обновлена.

## 12. Как отвечать пользователю

1. Отвечать по-русски.
2. Писать простым языком, без лишнего жаргона.
3. Сначала кратко дать план, затем работать.
4. В финале сообщать: что сделано; какие файлы изменены; какие проверки выполнены; что не проверено и почему; какие риски остались; что делать следующим шагом.
5. Не скрывать технические проблемы.
6. Не обсуждать юридические вопросы, если пользователь специально не просит.

## 13. Главный принцип

Asya 0.3 — это не интеграции и не голос. Это аккуратный слой памяти, личности, пространств, ленты памяти и прозрачности поверх multi-user фундамента 0.2.
