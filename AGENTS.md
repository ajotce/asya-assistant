# AGENTS.md

> Канонический документ для AI-агентов (Claude Code, Codex, opencode, Cursor и любых других), работающих с проектом `ajotce/asya-assistant`. `CLAUDE.md` и `OPENCODE.md` — тонкие обёртки, которые ссылаются на этот файл.

---

## Что такое Asya

Asya — персональный AI-ассистент. Версии 0.2–0.5 закрыты и смерджены в `main`. Сейчас проект входит в фазу **1.0 — Голос и публичный релиз**: переезд в облако, PostgreSQL, S3, мониторинг, бэкапы, OAuth (Google/Yandex), wake-word, открытая регистрация, плюс «отложенный продуктовый трансферт» из v0.5 — полноценные `document-templates` и `briefings/workflows`.

Полный план развития до 2.0+ — в `docs/roadmap.md`. Прочитай его перед началом работы.

## Текущая фаза и фокус

**Текущая фаза: 1.0 — Voice & Public Release.**

### Что входит в фазу 1.0

Из roadmap (раздел «Версия 1.0») плюс отложенный из v0.5 продуктовый трансферт:

**Инфраструктура и cloud-readiness:**
- 1.0.1 — Подготовка фазы и cloud-readiness audit (включая R1 и R6).
- 1.0.2 — Cloud-readiness fixes (нет локальных путей, всё через env-vars/абстракции, логи в stdout, `/healthz`).
- 1.0.3 — Миграция БД на PostgreSQL (включая замену `sqlite-vec` → `pgvector`).
- 1.0.4 — S3-совместимое хранилище для файлов (`boto3`/`aiobotocore`, MinIO для dev).
- 1.0.5 — Облачное развёртывание (Yandex Cloud / Selectel / AWS, Terraform, CI/CD).
- 1.0.6 — Мониторинг и алерты (Sentry, Prometheus+Grafana, централизованные логи).
- 1.0.7 — Резервное копирование (K6).

**Пользовательские фичи:**
- 1.0.8 — JSON-выгрузка пользователя и удаление учётки (K8).
- 1.0.9 — OAuth (Google, Yandex) — A6.
- 1.0.10 — Wake-word (G3) и режим «Слушаю» (G4).
- 1.0.11 — Открытая регистрация и onboarding.
- 1.0.12 — Public-facing сайт и юр. часть.

**Перенесено из v0.5 (deferred product-scope):**
- Полный product-scope **document-templates** (H1–H5): end-to-end DOCX/PDF pipeline, заполнение текстом и Vision (VIN/паспорт), конвертация DOCX→PDF, возврат в чат.
- Полный product-scope **briefings/workflows** (I2, I3): утренние/вечерние брифинги, real-time + briefings doc-режим.

**Стабилизация и релиз:**
- 1.0.13 — Стабилизация: нагрузочное тестирование, security audit (OWASP Top 10, RBAC, обработка секретов).
- 1.0.14 — Релиз `v1.0.0` и постмортем.

### Что **не** входит в 1.0

- **G5** (полноценный голосовой диалог с barge-in и interrupt) — это 2.0+/2.1.
- **F2, F5, F8** (Apple Calendar, Apple Reminders, Apple Mail через нативное приложение) — 2.0+.
- **D6** (общие пространства между пользователями) — 2.0+/2.2.
- **K7** (E2E-шифрование, zero-knowledge) — 2.0+ или never (после R7).
- **WhatsApp/Discord/VK/Max** — никогда.
- Apple OAuth (Sign in with Apple) — отложен в 2.0+.

Если задача требует чего-то из «не входит» — **остановись и спроси пользователя**. Не реализуй ничего без явного разрешения.

**Главные ветки:** `1.0-public` (рабочая ветка фазы), `main` (стабильный релиз).

## Установка и запуск

Команды (актуальные — в `README.md` и `Makefile`):

```bash
# Полный стек через Docker Compose (dev)
make dev                # = docker compose up --build
docker compose down

# Frontend
make build-frontend     # сборка frontend для бэкенда
make lint               # ESLint frontend

# Backend (локально, требует Python >=3.12)
make test               # backend pytest
# или контейнерные команды (если локально нет Python 3.12):
make backend-py312-pytest
make backend-py312-ruff
make backend-py312-mypy
make backend-py312-all

# Миграции БД
alembic upgrade head    # внутри backend-контейнера или локально

# CLI пользователей (admin)
python -m backend.cli user create --login alice --password securepass
```

Перед запуском убедись, что в `.env` заданы все обязательные переменные. Список — в `.env.example`.

С 1.0 в `.env` появятся новые блоки: `POSTGRES_*`, `S3_*`, `SENTRY_DSN`, `OAUTH_GOOGLE_*`, `OAUTH_YANDEX_*`, `WAKEWORD_*`. Все они должны быть представлены в `.env.example` как только соответствующая задача начата.

## Workflow разработки

### Маленькие шаги

Работай **маленькими завершёнными шагами**. Не пытайся сделать всю задачу одним коммитом. Каждый шаг — это:

1. Понятное изменение в одном-двух файлах.
2. Запуск проверок (тесты, линтер, типы).
3. Если всё зелёное — коммит с осмысленным сообщением.

Если шаг получается большим (более 200 строк изменений) — разбей его дальше.

### Перед началом задачи — план

После изучения релевантных файлов — выложи в чат **короткий план** (3–7 пунктов): что собираешься сделать, в каком порядке, какие файлы затронешь. Жди подтверждения пользователя.

Если задача очевидно простая (например, поправить опечатку в документации) — план не нужен, делай сразу.

### Проверки после каждого шага

После каждого шага запускай:

- Backend: `pytest backend/tests`, `ruff check`, `mypy backend` (если настроен).
- Frontend: `npm test`, `npm run lint`, `npm run typecheck`.
- Миграции: при изменении моделей БД — `alembic check` и применение на тестовой БД (Postgres-контейнер с 1.0).

Если проверка не запускается (например, сервис недоступен) — **прямо скажи об этом в ответе**, не пропускай молча.

### Финальный ответ

В финальном ответе кратко напиши:

- **Что сделано** — 1–3 предложения.
- **Какие файлы изменены** — список путей.
- **Какие проверки выполнены** — какие команды и что вернули.
- **Что не проверено и почему** — честно.
- **Следующий шаг** — что делать дальше.

Структура — в свободной форме, но все 5 пунктов должны быть.

## Параллельная работа нескольких агентов

В фазе 1.0 часть задач выполняется параллельно несколькими агентами (Codex / Claude Code / opencode) в разных ветках. Чтобы избежать конфликтов:

1. **Одна задача из roadmap → одна ветка → один агент.** Не запускай двух агентов на одну и ту же задачу.
2. **Имена веток:** `1.0/<task-id>-<short-name>`, например `1.0/1.0.3-postgres-migration`, `1.0/doc-templates-h1-models`.
3. **Префикс коммита** — короткий идентификатор задачи: `[1.0.3] PostgreSQL: add pgvector adapter`.
4. **PR в `1.0-public`**, не сразу в `main`. Merge в `main` — только после прохождения `docs/acceptance/v1.0.md` целиком.
5. **Перед началом — `git pull origin 1.0-public`**, чтобы видеть свежие изменения других агентов.
6. **Если задача затрагивает «общее» (`backend/db/`, `backend/core/config.py`, `docker-compose*.yml`, `Makefile`, `.env.example`)** — заранее предупреди в чате пользователя, чтобы он понимал, что эти файлы будут «горячими».
7. **Конфликты миграций Alembic:** строго одна `down_revision` цепочка. Если две ветки одновременно добавили миграции — мерджить через `alembic merge`.

Подробнее — в файле с пошаговыми промтами `docs/v1.0-prompts.md`.

## Запреты

### Безопасность

- **Никогда не коммитить** в репозиторий: `.env`, API-ключи, токены, пароли, master-key, приватные файлы, временные данные.
- **Никогда не логировать** секреты — даже сокращённо, даже в DEBUG.
- **Никогда не отправлять** реальные секреты в LLM как часть запросов.
- В `.env.example` — только примеры без реальных значений.
- При работе с любым секретом — сразу шифровать через слой `backend/security/encryption.py` (`encrypted_secrets`, Fernet, MASTER_ENCRYPTION_KEY).
- **JSON-выгрузка пользователя (K8) никогда не содержит токены интеграций** — это явное требование.
- При миграции на S3: bucket-policy запрещает публичный доступ; presigned URL только с TTL.

### Архитектура

- **Не делать** больших архитектурных изменений без явного обсуждения с пользователем. Большим считается: смена БД (на 1.0 как раз и происходит — но только в рамках утверждённой задачи 1.0.3), переезд на новый фреймворк, реорганизация модулей затрагивающая >5 файлов.
- **Не добавлять** production-зависимости без необходимости. Если зависимость нужна — коротко объясни в коммите или в `docs/decisions.md`, зачем она.
- **Не использовать** локальные пути к файлам или абсолютные пути на конкретной машине. Всё через env-vars или конфиг. Это требование задачи 1.0.2 (cloud-readiness).
- **Не хардкодить** ID, токены, имена ресурсов — только через конфиг или базу.

### Фичи вне фазы

- Не добавлять Apple-интеграции, G5 (barge-in), общие пространства, E2E, WhatsApp/Discord — это всё 2.0+ или never.
- Если в задаче есть намёк на такую фичу — **остановись и уточни** у пользователя.

### Действия с данными

- **Никаких массовых удалений** в коде или скриптах без флага явного подтверждения.
- При миграции с SQLite на PostgreSQL — миграционный скрипт обязан быть **идемпотентным**, с проверкой целостности и опцией dry-run.
- **При работе с production БД** — никаких прямых SQL-запросов на изменение данных без явного запроса пользователя; только через Alembic-миграции.
- При сомнениях — выбирай самый безопасный вариант и объясни выбор.

## Стиль кода

### Backend (Python)

- Python 3.12+. Не использовать deprecated синтаксис.
- Type hints везде. `mypy --strict` — цель.
- Форматирование: `ruff format`. Линтер: `ruff check`.
- Длина строки — 100 символов.
- Docstrings: Google-style для публичных функций и классов.
- Тесты: pytest, в `backend/tests/`. Один тест-файл на модуль кода.
- Async везде, где это уместно: FastAPI endpoints, IO-зависимые сервисы.

### Frontend (TypeScript / React)

- TypeScript strict mode.
- Форматирование: prettier. Линтер: ESLint.
- React функциональные компоненты + хуки. Никаких classes.
- Стили: существующая конвенция проекта (см. код).

### Общее

- Имена переменных и функций — описательные. Не `data`, `info`, `result` без контекста.
- Комментарии на русском или английском — последовательно в одном файле. По умолчанию — английский для технических, русский для пользовательских строк.
- Никакого «волшебного» кода без комментария почему так сделано.

## Тесты

### Что тестировать обязательно

- Все слои аутентификации (логин, проверка прав, изоляция данных между пользователями).
- OAuth-flow (PKCE, refresh, revoke) для каждой интеграции с моками.
- Все слои шифрования (encrypt/decrypt round-trip, обработка ошибок).
- Все CLI-команды.
- Миграции БД (upgrade и downgrade на Postgres и SQLite, пока поддерживаются оба).
- Изоляцию пользователей (user A не видит данные user B по ID).
- Скрипт миграции SQLite → PostgreSQL (тестовая БД с фикстурами).
- S3-абстракция: round-trip загрузка/чтение/удаление в MinIO.

### Что покрытие важно поддерживать

- Сервисы (`SessionStore`, `ChatService`, `ActionRouter`, и т.д.) — unit-тесты.
- API endpoints — integration-тесты через TestClient.
- Frontend — компоненты с критической логикой.

### Что не нужно тестировать

- Тривиальные геттеры/сеттеры.
- Сторонние библиотеки.
- Логи и трейсинг.

## Документация

При изменении поведения проекта — обновляй документацию **в том же коммите** или сразу в следующем:

- Изменился API → обнови `docs/api.md`.
- Изменилась архитектура → обнови `docs/architecture.md`.
- Принято важное архитектурное решение → запиши в `docs/decisions.md` (формат ADR: контекст, решение, последствия).
- Изменился процесс разработки → обнови `docs/development.md`.
- Завершена крупная задача — добавь запись в `docs/development-log.md`.
- Изменения в интеграциях → `docs/integrations.md`.
- Изменения в инфраструктуре/деплое → `docs/deployment.md`.
- Документы шаблонов и брифингов (новые) — `docs/templates.md`, `docs/briefings.md`.

Документация на русском языке (по предпочтению пользователя). Технические термины можно оставлять на английском, если у них нет устоявшегося русского аналога.

## Pull request / merge

- Каждая фаза (0.4, 0.5, 1.0, ...) живёт в своей ветке.
- В пределах фазы — небольшие коммиты с осмысленными сообщениями.
- PR из `1.0/<task>` → `1.0-public`. Финальный PR `1.0-public` → `main`.
- Перед merge ветки фазы в `main`:
  1. Все тесты зелёные (backend + frontend).
  2. Все пункты `docs/acceptance/v1.0.md` пройдены.
  3. Документация обновлена.
  4. Нет TODO без issue в трекере.
  5. Нет закомментированного кода.
  6. Security audit пройден без критических замечаний.
  7. Нагрузочный тест выполнен, отчёт в `docs/development-log.md`.

## Уточнения и эскалация

### Когда задавать уточнения

- Без ответа продолжать **опасно** (можно случайно удалить данные, поломать продакшен, скомпрометировать секреты).
- Без ответа продолжать **невозможно** (не понятно что делать).
- Задача требует чего-то из «не входит в текущую фазу».
- Найден неожиданный архитектурный конфликт.
- Миграционный скрипт может затронуть продовые данные.

### Когда не задавать уточнения

- Можно выбрать самый безопасный минимальный вариант — выбирай и объясняй в ответе.
- Это деталь стиля кода — следуй существующим конвенциям проекта.
- Это вопрос форматирования — следуй ruff/prettier.
