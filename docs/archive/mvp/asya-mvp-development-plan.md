# Пошаговый план разработки Asya MVP

**Проект:** Asya / Ася — персональный ИИ-ассистент  
**Тип документа:** пошаговый план разработки для Codex / Claude Code  
**Основание:** `asya-mvp-tech-spec.md`  
**Версия:** 1.0  
**Дата:** 25 апреля 2026  
**Цель:** поэтапно разработать MVP без выхода за рамки ТЗ

---

## 1. Главный принцип разработки

Разработка должна идти короткими проверяемыми этапами. После каждого этапа проект должен запускаться, а изменения должны быть проверены тестами или ручной проверкой.

Важно: не пытаться сразу сделать весь проект целиком. Codex / Claude Code должен двигаться постепенно:

1. создать каркас проекта;
2. запустить backend;
3. запустить frontend;
4. связать frontend и backend;
5. подключить VseLLM;
6. добавить streaming;
7. добавить настройки;
8. добавить сессии;
9. добавить файлы;
10. добавить embeddings и временную векторную базу;
11. добавить страницу состояния;
12. добавить тесты и документацию;
13. провести финальную приёмку.

---

## 2. Что нельзя добавлять в MVP

Во время разработки запрещено добавлять функции, которых нет в ТЗ.

Не добавлять:

- Todoist;
- Google Calendar;
- Apple Calendar;
- Google Contacts;
- Apple Contacts;
- Bitrix24;
- веб-поиск;
- голосовой ввод;
- озвучку ответов;
- долгосрочную память;
- историю чатов;
- список чатов;
- несколько пользователей;
- регистрацию;
- PIN-код;
- полноценную авторизацию;
- production-деплой на VPS;
- reverse proxy;
- тёмную тему;
- сложные UI-фреймворки;
- облачную векторную базу;
- хранение API-ключей во frontend.

Если Codex / Claude Code предлагает добавить что-то из этого списка, нужно отказаться и вернуться к MVP.

## 3. Целевой стек

### 3.1. Frontend

Использовать:

- React;
- Vite;
- TypeScript;
- PWA;
- CSS Modules или простой CSS;
- Fetch API;
- потоковое чтение ответа через `ReadableStream`.

### 3.2. Backend

Использовать:

- Python 3.12+;
- FastAPI;
- Uvicorn;
- Pydantic;
- OpenAI-compatible клиент для VseLLM;
- SQLite;
- ChromaDB или аналогичную локальную временную векторную базу;
- Docker;
- Docker Compose;
- pytest.

### 3.3. Файлы

Для обработки файлов использовать:

- PDF: PyMuPDF;
- DOCX: python-docx;
- XLSX: openpyxl;
- изображения: Pillow для валидации, vision-модель для анализа.

## 5. Этап 0. Подготовка проекта

### 5.1. Цель этапа

Создать пустой репозиторий с правильной структурой, базовыми файлами и правилами разработки.

### 5.2. Задачи

1. Создать структуру папок `frontend`, `backend`, `docs`.
2. Создать `.gitignore`.
3. Создать `.env.example`.
4. Создать `README.md` с кратким описанием проекта.
5. Создать `docs/decisions.md` и зафиксировать ключевые решения MVP.
6. Создать `docker-compose.yml` с backend-сервисом.
7. Создать `Makefile` или описать команды запуска в README.

### 5.3. `.env.example`

Минимальные переменные:

```env
APP_ENV=local
ASYA_HOST=127.0.0.1
ASYA_PORT=8000

VSELLM_API_KEY=
VSELLM_BASE_URL=https://api.vsellm.ru/v1
DEFAULT_CHAT_MODEL=
DEFAULT_EMBEDDING_MODEL=

SQLITE_PATH=/app/data/asya.sqlite3
TMP_DIR=/app/tmp
MAX_FILES_PER_MESSAGE=10
MAX_FILE_SIZE_MB=256
```

### 5.4. Проверка готовности этапа

Этап считается готовым, если:

- структура проекта создана;
- `.env.example` есть;
- `.env` добавлен в `.gitignore`;
- README объясняет назначение проекта;
- в `docs/decisions.md` зафиксированы ограничения MVP.

## 6. Этап 1. Backend-каркас FastAPI

### 6.1. Цель этапа

Запустить минимальный backend в Docker с FastAPI, OpenAPI/Swagger и health-check.

### 6.2. Задачи

1. Создать backend-приложение FastAPI.
2. Настроить `pyproject.toml`.
3. Добавить конфигурацию через Pydantic settings.
4. Добавить базовое логирование.
5. Добавить endpoint `GET /api/health`.
6. Добавить Dockerfile backend.
7. Настроить запуск через Docker Compose.
8. Проверить Swagger UI.

### 6.3. Минимальные backend endpoints

```text
GET /api/health
```

Ответ должен содержать:

```json
{
  "status": "ok",
  "version": "0.1.0",
  "environment": "local",
  "vsellm": {
    "api_key_configured": false,
    "base_url": "https://api.vsellm.ru/v1"
  }
}
```

### 6.4. Проверка готовности этапа

Этап считается готовым, если:

- `docker compose up --build` запускает backend;
- `http://localhost:8000/api/health` возвращает JSON;
- `http://localhost:8000/docs` открывает Swagger;
- backend читает `.env`;
- API-ключ не выводится в ответах и логах.

- `docker compose up --build` запускает backend;
- `http://localhost:8000` открывает frontend;

## 10. Этап 5. Интеграция со списком моделей VseLLM

### 10.1. Цель этапа

Сделать автоматическую загрузку списка моделей через backend.

### 10.2. Backend задачи

1. Создать сервис `vsellm_client.py`.
2. Реализовать запрос к:

```text
GET /v1/models
```

3. Добавить endpoint:

```text
GET /api/models
```

4. Нормализовать ответ в формат:

```json
[
  {
    "id": "model-id",
    "name": "model-id",
    "description": null,
    "context_window": null,
    "input_price": null,
    "output_price": null,
    "supports_vision": null,
    "raw": {}
  }
]
```

5. Если VseLLM не отдаёт описание/цену, возвращать только ID.
6. Обработать ошибки 401, 403, 429, 500, timeout.

### 10.3. Frontend задачи

1. На странице настроек добавить кнопку `Обновить список моделей`.
2. Показать список моделей.
3. Дать выбрать модель.
4. Сохранить выбранную модель в settings.
5. Если список моделей не загрузился, показать понятную ошибку.

### 10.4. Проверка готовности этапа

Этап считается готовым, если:

- frontend получает список моделей через `/api/models`;
- пользователь может выбрать модель;
- выбранная модель сохраняется;
- если API недоступен, UI не ломается.

```text
POST /api/session
GET /api/session/{session_id}
DELETE /api/session/{session_id}
```

5. При удалении сессии очищать:
   - сообщения;

### 11.3. Frontend задачи

1. При первом открытии создать сессию.
2. Хранить `session_id` в памяти frontend или sessionStorage.
3. Добавить кнопку `Очистить сессию`.
4. После очистки создавать новую сессию или очищать текущую.

### 11.4. Проверка готовности этапа

Этап считается готовым, если:

- сессия создаётся;
- сообщения можно привязать к сессии;
- сессия очищается;
- после очистки контекст не используется;
- история не сохраняется как список чатов.

3. Реализовать запрос к:

```text
POST /v1/chat/completions
```

4. Использовать выбранную модель из settings.
5. Сохранять user/assistant messages в текущую сессию.
6. Добавить временный endpoint:

```text
POST /api/chat
```

7. Обработать ошибки модели.

### 12.3. Frontend задачи

1. Отправлять сообщение на backend.
2. Показывать ответ модели.
3. Показывать loading-состояние.
4. Показывать ошибку, если запрос не удался.

### 12.4. Проверка готовности этапа

Этап считается готовым, если:

- пользователь может отправить сообщение;
- ответ приходит от VseLLM;
- выбранная модель используется;
- системный промт используется;
- сообщения остаются в текущей сессии;
- после очистки сессии контекст исчезает.

## 13. Этап 8. Streaming-ответы

### 13.1. Цель этапа

Заменить обычный ответ на потоковый вывод текста.

### 13.2. Backend задачи

1. Реализовать endpoint:

```text
POST /api/chat/stream
```

2. Выбрать формат streaming:
   - SSE;
   - или NDJSON.
3. Передавать токены/части ответа постепенно.
4. По завершении сохранять полный assistant message в сессию.
5. Передавать usage, если API его возвращает.
6. Корректно закрывать поток при ошибке.

### 13.3. Frontend задачи

1. Читать поток через `ReadableStream`.
2. Показывать текст постепенно.
3. Показывать индикатор генерации.
4. После завершения фиксировать ответ в списке сообщений.
5. При ошибке показывать понятное сообщение.

### 13.4. Проверка готовности этапа

Этап считается готовым, если:

- ответ появляется постепенно;
- длинный ответ не блокирует UI;
- после завершения ответ сохранён в текущей сессии;
- при ошибке поток закрывается корректно;
- frontend не зависает.

### 14.4. Проверка готовности этапа

- пользователь может отредактировать сообщение;

## 16. Этап 11. Загрузка файлов

### 16.1. Цель этапа

Реализовать загрузку файлов в рамках текущей временной сессии.

### 16.2. Backend задачи

1. Создать `file_service.py`.
2. Создать endpoints:

```text
POST /api/files
GET /api/files/{file_id}
DELETE /api/files/{file_id}
```

3. Требовать `session_id` при загрузке.
4. Проверять лимиты:
   - максимум 10 файлов за одно сообщение;
   - максимум 256 МБ на файл.
5. Проверять типы:
   - PDF;
   - DOCX;
   - XLSX;
   - PNG;
   - JPG/JPEG;
   - WEBP.
6. Сохранять файлы во временную директорию с привязкой к session_id.
7. Не хранить файлы после очистки сессии.

### 16.3. Frontend задачи

1. Добавить кнопку прикрепления файлов.
2. Показать список выбранных файлов перед отправкой.
3. Показывать прогресс загрузки, если возможно.
4. Показывать ошибки лимитов и форматов.
5. После отправки сообщения связывать uploaded file ids с chat request.

### 16.4. Проверка готовности этапа

Этап считается готовым, если:

- файлы загружаются;
- неподдерживаемые форматы отклоняются;
- файлы больше 256 МБ отклоняются;
- больше 10 файлов за сообщение отклоняется;
- очистка сессии удаляет файлы.

## 17. Этап 12. Извлечение текста из PDF/DOCX/XLSX

### 17.1. Цель этапа

Научить backend извлекать текст из поддерживаемых документов.

### 17.2. Backend задачи

1. Создать `extraction_service.py`.
2. Для PDF использовать PyMuPDF.
3. Для DOCX использовать python-docx.
4. Для XLSX использовать openpyxl.
5. Для каждого документа возвращать унифицированную структуру:

```json
{
  "file_id": "...",
  "filename": "...",
  "text_blocks": [
    {
      "source": "page 1",
      "text": "..."
    }
  ]
}
```

6. Если PDF без текстового слоя — вернуть понятную ошибку.
7. OCR не делать.

### 17.3. Проверка готовности этапа

Этап считается готовым, если:

- текст из PDF извлекается;
- текст из DOCX извлекается;
- текст из XLSX извлекается;
- пустой/сканированный PDF обрабатывается без падения;
- ошибки понятны пользователю.

## 18. Этап 13. Chunking документов

### 18.1. Цель этапа

Разбивать извлечённый текст на фрагменты для embeddings и retrieval.

### 18.2. Backend задачи

1. Реализовать chunking.
2. Размер chunk: ориентировочно 800–1500 токенов или эквивалент по символам.
3. Overlap: 100–200 токенов или эквивалент.
4. Сохранять metadata:
   - file_id;
   - filename;
   - page/sheet;
   - chunk_index;
   - session_id.
5. Не вставлять огромные документы целиком в контекст модели.

### 18.3. Проверка готовности этапа

Этап считается готовым, если:

- большой документ разбивается на chunks;
- chunks имеют metadata;
- chunks привязаны к сессии;
- chunking не ломает обработку маленьких файлов.

## 19. Этап 14. Embeddings и временная векторная база

### 19.1. Цель этапа

Реализовать поиск по документам через embeddings + локальную временную векторную базу.

### 19.2. Backend задачи

1. Создать `embedding_service.py`.
2. Создать `vector_store.py`.
3. Использовать VseLLM embeddings endpoint:

```text
POST /v1/embeddings
```

4. Embedding-модель брать из settings или `.env`.
5. Создавать отдельную collection на session_id.
6. Добавлять chunks в collection.
7. При очистке сессии удалять collection.
8. Обработать ошибки embeddings.

### 19.3. Retrieval задачи

При вопросе пользователя:

1. Проверить, есть ли файлы в сессии.
2. Получить embedding вопроса.
3. Найти top-k chunks.
4. Добавить релевантные chunks в контекст модели отдельным блоком.
5. Попросить модель отвечать по файлам, если вопрос связан с ними.

### 19.4. Проверка готовности этапа

Этап считается готовым, если:

- chunks индексируются;
- вопрос по документу находит релевантные chunks;
- ответ модели использует найденные chunks;
- очистка сессии удаляет collection;
- ошибка embeddings не ломает весь чат.
