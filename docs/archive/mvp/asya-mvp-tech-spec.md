# Техническое задание: Asya MVP

**Проект:** персональный ИИ-ассистент Asya / Ася  
**Тип документа:** финальное техническое задание для разработки через OpenAI Codex / Claude Code  
**Версия:** 1.0  
**Дата:** 25 апреля 2026  
**Язык интерфейса:** русский  
**Целевая платформа:** Web / PWA для Safari на iPhone и Safari/Chrome на Mac  
**Основной режим развёртывания MVP:** локально на MacBook, backend в Docker  

---

## 1. Краткое описание проекта

Asya — персональный ИИ-ассистент в формате веб-чата. В MVP Asya должна работать как текстовый чат-бот с возможностью выбора модели, настройки системного промта, потоковой генерации ответов и работы с файлами в рамках текущей временной сессии.

MVP не должен превращаться в полноценную систему задач, CRM, календарей или долгосрочной памяти. Главная цель MVP — получить стабильный рабочий чат-интерфейс с VseLLM, который можно использовать с Mac и iPhone, а затем постепенно расширять.

---

## 2. Цели MVP

### 2.1. Основная цель

Создать минимально необходимую, но архитектурно расширяемую версию Asya:

- текстовый чат;
- выбор модели через настройки;
- интеграция с VseLLM через OpenAI-compatible API;
- потоковая генерация ответа;
- временный контекст одной backend-сессии;
- загрузка и анализ файлов;
- редактируемый системный промт;
- страница состояния системы;
- локальный запуск backend в Docker;
- frontend в формате PWA.

### 2.2. Что должно быть удобно для пользователя

Пользователь не пишет код самостоятельно. Проект должен быть устроен так, чтобы его можно было разрабатывать и поддерживать через Codex / Claude Code.

Поэтому в проекте должны быть:

- понятная структура папок;
- строгая типизация на frontend;
- автотесты для критичных частей;
- документация, обновляемая по мере разработки;
- `.env.example`;
- Docker Compose;
- OpenAPI/Swagger-документация backend API;
- понятные команды запуска.

## 4. Пользователь и сценарии использования

### 4.1. Пользователь

В MVP есть один пользователь — владелец проекта.

Система не должна содержать регистрацию, роли, группы пользователей или разграничение прав.

### 4.2. Основные сценарии

#### Сценарий 1. Обычный чат

1. Пользователь открывает Asya.
2. Пишет сообщение.
3. Backend отправляет запрос в выбранную модель VseLLM.
4. Ответ отображается потоково.
5. Сообщения остаются в текущей временной сессии.
6. После очистки сессии или перезапуска backend контекст теряется.

#### Сценарий 2. Выбор модели

1. Пользователь открывает настройки.
2. Backend запрашивает список моделей через VseLLM API.
3. Frontend показывает список моделей.
4. Если API отдаёт только ID модели, показывается только ID.
5. Пользователь выбирает модель.
6. Выбор применяется глобально для текущего пользователя и всех запросов.

#### Сценарий 3. Изменение системного промта

1. Пользователь открывает настройки.
2. Редактирует системный промт Asya.
3. Сохраняет настройки.
4. Все новые запросы используют обновлённый системный промт.

#### Сценарий 4. Загрузка файлов

1. Пользователь прикрепляет до 10 файлов к сообщению.
2. Backend валидирует типы и размер.
3. Backend извлекает текст или готовит изображение для vision-модели.
4. Текстовые документы разбиваются на фрагменты.
5. Для фрагментов создаются embeddings.
6. Временная векторная база связывает фрагменты с текущей сессией.
7. При вопросе по файлам backend выбирает релевантные фрагменты и добавляет их в контекст модели.
8. Файлы и индекс живут только в рамках временной сессии.

#### Сценарий 5. Редактирование сообщения

1. Пользователь редактирует своё последнее или предыдущее сообщение.
2. Система пересобирает контекст начиная с изменённого сообщения.
3. Старый ответ после отредактированного сообщения помечается как устаревший или удаляется.
4. Backend генерирует новый ответ.

#### Сценарий 6. Проверка состояния Asya

1. Пользователь открывает страницу «Состояние Asya».
2. Frontend запрашивает `/api/health`.
3. Пользователь видит статус backend, VseLLM API, выбранной модели, файлового модуля и временной сессии.

## 5. Технологический стек

### 5.1. Frontend

Использовать:

- React;
- Vite;
- TypeScript;
- PWA;
- CSS Modules или обычный CSS с понятной структурой;
- Fetch API для общения с backend;
- потоковое чтение ответа через `ReadableStream`.

Не использовать в MVP без необходимости:

- Next.js;
- Redux;
- сложные UI-фреймворки;
- серверный рендеринг;
- сложные дизайн-системы.

### 5.2. Backend

Использовать:

- Python 3.12+;
- FastAPI;
- Uvicorn;
- Pydantic;
- OpenAI Python SDK или совместимый HTTP-клиент;
- Docker;
- Docker Compose;
- SQLite для настроек, usage и служебных данных;
- ChromaDB или аналогичную локальную векторную базу для временного поиска по файлам;
- PyMuPDF для PDF;
- python-docx для DOCX;
- openpyxl для XLSX;
- Pillow для базовой проверки изображений;
- pytest для тестов backend.

### 5.3. Векторная база

Для MVP использовать локальную временную векторную базу.

Рекомендуемый вариант реализации:

- ChromaDB в локальном режиме внутри backend-контейнера;
- отдельная collection на каждую backend-сессию;
- collection удаляется при очистке сессии;
- при перезапуске backend временный контекст может быть потерян;
- векторная база не используется как долговременная память.

Если Codex / Claude Code предложит более простую и стабильную альтернативу, можно использовать её при условии сохранения требований:

- всё локально;
- без облачной векторной базы;
- данные только временные;
- есть интерфейс-обёртка, чтобы позже заменить реализацию.

## 6. Архитектура

### 6.1. Общая схема

```text
Пользователь
  ↓
Frontend PWA: React + Vite + TypeScript
  ↓ HTTP локально
Backend API: FastAPI в Docker
  ↓
VseLLM OpenAI-compatible API
```

Дополнительные локальные компоненты backend:

```text
FastAPI backend
  ├─ Settings storage: SQLite
  ├─ Temporary session memory
  ├─ Temporary file storage
  ├─ File text extraction
  ├─ Embeddings client
  ├─ Temporary vector store
  ├─ Usage tracker
  └─ Health-check
```

### 6.2. Важное решение по GitHub Pages и локальному backend

В MVP backend разворачивается только локально.

GitHub Pages работает по HTTPS. Локальный backend на MacBook обычно доступен по HTTP. Из-за этого возможны проблемы с запросами из HTTPS-frontend к локальному HTTP-backend.

Поэтому для MVP нужно реализовать два режима frontend:

#### Режим 1. Локальный основной режим MVP

Backend раздаёт собранный frontend сам.

```text
FastAPI backend
  ├─ /api/*        backend API
  └─ /*            static frontend build
```

Пользователь открывает Asya через локальный адрес backend.

Для Mac:

```text
http://localhost:8000
```

Для iPhone в той же Wi‑Fi сети:

```text
http://<MACBOOK_LOCAL_IP>:8000
```

#### Режим 2. GitHub Pages для будущего использования

Frontend должен уметь собираться как static build для GitHub Pages, но этот режим считается вспомогательным до тех пор, пока backend не будет доступен по HTTPS.

В MVP GitHub Pages не является основным способом работы с локальным backend.

### 6.3. PWA и HTTPS

PWA должна иметь:

- `manifest.webmanifest`;
- app name: `Asya` или `Ася`;
- icons;
- theme color;
- service worker;
- offline cache только для frontend shell.

Важно:

- chat-функции требуют backend и интернет-доступ к VseLLM;
- offline-чат в MVP не нужен;
- полноценная работа service worker на iPhone требует secure context;
- для локального iPhone-доступа по HTTP возможны ограничения PWA-функций;
- это не должно блокировать обычную работу чата в Safari.

## 7. Интеграция с VseLLM

### 7.1. Базовые параметры

Использовать VseLLM как OpenAI-compatible API.

Базовый URL по умолчанию:

```text
https://api.vsellm.ru/v1
```

API-ключ хранить только на backend в `.env`.

Пример переменных окружения:

```env
VSELLM_API_KEY=sk-...
VSELLM_BASE_URL=https://api.vsellm.ru/v1
DEFAULT_CHAT_MODEL=openai/gpt-5
DEFAULT_EMBEDDING_MODEL=openai/text-embedding-3-small
```

### 7.2. Получение моделей

Backend должен иметь endpoint, который получает список моделей через VseLLM:

```text
GET /v1/models
```

Frontend не должен напрямую обращаться к VseLLM.

Frontend обращается только к backend:

```text
GET /api/models
```

Backend возвращает нормализованный список:

```json
[
  {
    "id": "openai/gpt-5",
    "name": "openai/gpt-5",
    "description": null,
    "context_window": null,
    "input_price": null,
    "output_price": null,
    "supports_vision": null,
    "raw": {}
  }
]
```

Если VseLLM API не отдаёт описание, цену или размер контекста, показывать только ID модели.

### 7.3. Chat Completions

Для обычного чата использовать:

```text
POST /v1/chat/completions
```

Требования:

- использовать выбранную модель;
- передавать системный промт;
- передавать сообщения текущей временной сессии;
- включать потоковую генерацию;
- обрабатывать ошибки 400/401/403/404/429/500/502;
- при ошибке модели показывать понятное сообщение пользователю.

### 7.4. Embeddings

Для поиска по загруженным документам использовать embeddings:

```text
POST /v1/embeddings
```

Требования:

- модель embeddings задаётся через `.env`;
- embeddings используются только для временных файлов текущей сессии;
- embeddings не используются для долгосрочной памяти пользователя;
- при ошибке embeddings модуль файлов должен вернуть понятное сообщение.

### 7.5. Vision

Изображения поддерживаются только если выбранная chat-модель умеет работать с изображениями.

Если пользователь загрузил изображение, а выбранная модель не поддерживает vision или API вернул ошибку:

- не падать;
- показать понятное сообщение;
- предложить выбрать модель с поддержкой изображений.

OCR отдельным модулем в MVP не требуется.

## 8. Сессии, память и хранение данных

### 8.1. История чатов

История чатов в MVP не сохраняется.

Не должно быть:

- списка прошлых диалогов;
- поиска по истории;
- архива чатов;
- долговременного журнала сообщений.

### 8.2. Временная backend-сессия

При первом открытии frontend получает или создаёт `session_id`.

Сессия хранит:

- текущие сообщения;
- загруженные файлы;
- извлечённые текстовые фрагменты;
- временные embeddings;
- usage за текущую сессию.

Хранение — временное, в памяти backend и/или во временной директории контейнера.

После перезапуска backend сессия может быть потеряна. Это допустимо для MVP.

### 8.3. Очистка сессии

В интерфейсе должна быть кнопка:

```text
Очистить сессию
```

После очистки:

- удаляются сообщения текущей сессии;
- удаляются загруженные файлы;
- удаляются chunks;
- удаляются embeddings;
- удаляется collection векторной базы;
- usage текущей сессии сбрасывается или архивируется как агрегированная статистика без сообщений.

### 8.4. Долговременная память

В MVP долговременная память запрещена.

Asya не должна:

- автоматически сохранять факты о пользователе;
- предлагать сохранить воспоминание;
- создавать профиль пользователя;
- использовать embeddings как персональную память;
- помнить информацию после завершения/очистки сессии.

## 9. Настройки

### 9.1. Какие настройки должны быть в MVP

Страница настроек должна содержать:

1. Имя ассистента.
2. Выбранная chat-модель.
3. Системный промт.
4. Base URL VseLLM, только для просмотра или редактирования через backend settings, но без отображения API-ключа.
5. Статус API-ключа: настроен / не настроен.
6. Embedding-модель, можно только как advanced-настройка.
7. Кнопка обновления списка моделей.
8. Кнопка проверки подключения к VseLLM.

### 9.2. API-ключи

API-ключ VseLLM не вводится через frontend в MVP.

Причина: в MVP нет авторизации, поэтому ввод и хранение ключей через интерфейс небезопасны.

Ключ должен храниться в `.env` backend.

Frontend может показывать только статус:

```text
API-ключ VseLLM: настроен
```

или

```text
API-ключ VseLLM: не найден
```

### 9.3. Системный промт по умолчанию

Системный промт по умолчанию:

```text
Ты — Asya, персональный ИИ-ассистент пользователя. Общайся на русском языке, но можешь понимать и использовать английский, когда это нужно. Стиль общения: деловой, дружелюбный, понятный. Ты можешь быть личным помощником, деловым ассистентом, секретарём, другом и универсальным помощником.

Выполняй задачу максимально полезно. Если информации достаточно — не задавай лишних уточняющих вопросов и действуй по best effort, явно указывая важные допущения. Если без уточнения невозможно выполнить задачу качественно, есть риск ошибки, или есть несколько принципиально разных вариантов — сначала задай уточняющий вопрос.

Не будь проактивной: не присылай напоминания и не инициируй действия сама. В MVP у тебя нет долговременной памяти, нет истории прошлых чатов и нет доступа к внешним интеграциям. Работай только с текущей сессией и файлами, которые пользователь загрузил в этой сессии.
```

Пользователь должен иметь возможность редактировать этот промт в настройках.

## 11. Работа с файлами

### 11.1. Поддерживаемые типы

MVP должен поддерживать загрузку:

- PDF;
- DOCX;
- XLSX;
- PNG;
- JPG/JPEG;
- WEBP.

### 11.2. Лимиты

- Максимум 10 файлов за одно сообщение.
- Максимальный размер каждого файла: 256 МБ.
- Если файл превышает лимит, отклонить загрузку и показать понятную ошибку.

### 11.3. Обработка PDF

Для PDF:

- извлечь текст;
- сохранить структуру по страницам, если возможно;
- сформировать chunks;
- создать embeddings;
- сохранить ссылку chunk → файл → страница.

Если PDF сканированный и текста нет:

- OCR в MVP не требуется;
- показать сообщение, что текст не найден;
- предложить использовать PDF с текстовым слоем или vision-модель, если это применимо.

### 11.4. Обработка DOCX

Для DOCX:

- извлечь текст параграфов;
- извлечь таблицы, если возможно;
- сформировать chunks;
- создать embeddings.

### 11.5. Обработка XLSX

Для XLSX:

- извлечь названия листов;
- извлечь данные таблиц;
- преобразовать содержимое в понятный текстовый формат;
- сохранить привязку chunk → файл → лист → диапазон строк, если возможно.

Важно: нельзя пытаться полностью вставлять огромный XLSX в контекст модели. Нужно использовать chunking + retrieval.

### 11.6. Обработка изображений

Для изображений:

- проверить MIME type;
- проверить размер;
- передавать изображение в chat-модель как multimodal input, если выбранная модель поддерживает vision;
- если модель не поддерживает vision, вернуть понятную ошибку.

### 11.7. Chunking

Рекомендации:

- размер chunk: 800–1500 токенов или эквивалент по символам;
- overlap: 100–200 токенов;
- сохранять metadata: file_id, filename, page/sheet, chunk_index;
- retrieval должен возвращать top-k релевантных chunks.

### 11.8. Retrieval

При вопросе пользователя backend должен:

1. Определить, есть ли в сессии файлы.
2. Если есть файлы, получить embedding вопроса.
3. Найти релевантные chunks во временной векторной базе.
4. Добавить найденные chunks в контекст модели как отдельный блок.
5. В системном или developer-контексте указать модели, что она должна отвечать на основе загруженных файлов, если вопрос связан с ними.

## 12. Usage и токены

### 12.1. Что считать

Backend должен пытаться сохранять:

- input tokens;
- output tokens;
- total tokens;
- модель;
- время запроса;
- длительность ответа;
- статус успех/ошибка.

### 12.2. Источник данных

Основной источник — поле `usage` из ответа VseLLM/OpenAI-compatible API.

Если при streaming API usage не возвращается:

- не блокировать работу;
- показывать `нет данных`;
- не делать сложную собственную оценку токенов в MVP, если это усложняет разработку.

### 12.3. Стоимость

Если VseLLM API не отдаёт цену модели, не рассчитывать стоимость.

В MVP показывать только:

- количество токенов, если доступно;
- ID модели;
- количество запросов;
- ошибки.

## 13. Backend API

Все backend endpoints должны начинаться с `/api`.

### 13.1. Health

```text
GET /api/health
```

Пример ответа:

```json
{
  "status": "ok",
  "version": "0.1.0",
  "uptime_seconds": 1234,
  "vsellm": {
    "api_key_configured": true,
    "base_url": "https://api.vsellm.ru/v1",
    "reachable": true
  },
  "model": {
    "selected": "openai/gpt-5"
  },
  "files": {
    "enabled": true
  },
  "embeddings": {
    "enabled": true,
    "model": "openai/text-embedding-3-small"
  }
}
```

### 13.2. Models

```text
GET /api/models
```

Возвращает список моделей из VseLLM.

### 13.3. Settings

```text
GET /api/settings
PUT /api/settings
```

Settings включают:

- assistant_name;
- selected_model;
- system_prompt;
- base_url;
- embedding_model.

API-ключ никогда не возвращается frontend.

### 13.4. Session

```text
POST /api/session
GET /api/session/{session_id}
DELETE /api/session/{session_id}
```

Назначение:

- создать сессию;
- получить состояние сессии;
- очистить сессию.

### 13.5. Chat streaming

```text
POST /api/chat/stream
```

Пример запроса:

```json
{
  "session_id": "uuid",
  "message": "Привет, Ася",
  "attachments": ["file_id_1"],
  "edited_from_message_id": null
}
```

Ответ должен быть потоковым.

Рекомендуемый формат:

```text
event: token
data: {"text":"Привет"}

event: token
data: {"text":", чем"}

event: done
data: {"message_id":"...","usage":null}
```

Можно использовать NDJSON вместо SSE, если реализация frontend будет проще. Главное — потоковый вывод в интерфейсе.

### 13.6. Files

```text
POST /api/files
GET /api/files/{file_id}
DELETE /api/files/{file_id}
```

Загрузка файла должна требовать `session_id`.

Файл не должен существовать вне временной сессии.

### 13.7. Usage

```text
GET /api/usage/session/{session_id}
```

Возвращает usage текущей сессии.

## 14. Ошибки и UX сообщений

### 14.1. Ошибки VseLLM

Обработать минимум:

- 400 Bad Request;
- 401 Unauthorized;
- 403 Forbidden;
- 404 Model Not Found;
- 429 Too Many Requests;
- 500 Internal Server Error;
- 502 Bad Gateway;
- timeout.

Пользовательские сообщения должны быть понятными.

Примеры:

```text
Не удалось подключиться к VseLLM. Проверь API-ключ и подключение к интернету.
```

```text
Модель не найдена. Обнови список моделей в настройках и выбери модель заново.
```

```text
Сервис временно ограничил частоту запросов. Попробуй повторить позже.
```

### 14.2. Ошибки файлов

Обработать:

- неподдерживаемый формат;
- превышение размера;
- превышение количества файлов;
- ошибка извлечения текста;
- пустой PDF без текстового слоя;
- ошибка embeddings;
- ошибка vector store.

### 14.3. Ошибки frontend

Frontend не должен падать белым экраном.

Должны быть:

- понятные сообщения об ошибке;
- loading-состояния;
- retry-кнопка там, где это уместно;
- сохранение текста, который пользователь набрал, если запрос не отправился.

## 15. Безопасность

### 15.1. Авторизация

В MVP авторизация не нужна.

Причина: backend используется только локально.

### 15.2. Ограничение доступа

По умолчанию backend должен запускаться в безопасном локальном режиме.

Рекомендуемые режимы:

#### Mac-only

```env
ASYA_HOST=127.0.0.1
```

Доступ только с MacBook:

```text
http://localhost:8000
```

#### Local network для iPhone

```env
ASYA_HOST=0.0.0.0
```

Доступ с iPhone в той же Wi‑Fi сети:

```text
http://<MACBOOK_LOCAL_IP>:8000
```

В документации нужно явно предупредить:

```text
Режим ASYA_HOST=0.0.0.0 открывает backend в локальной сети. Использовать только в доверенной Wi‑Fi сети.
```

### 15.3. API-ключи

API-ключ VseLLM хранится только в backend `.env`.

Запрещено:

- хранить API-ключ в frontend;
- отдавать API-ключ через API;
- писать API-ключ в логи;
- коммитить `.env` в Git.

### 15.4. Логи

Логи могут содержать текст сообщений пользователя, так как пользователь разрешил это для MVP.

Но запрещено логировать:

- API-ключи;
- полные переменные окружения;
- системные секреты;
- бинарное содержимое файлов.

## 16. Логирование

Backend должен логировать:

- старт приложения;
- ошибки подключения к VseLLM;
- ошибки файлового парсинга;
- ошибки embeddings;
- ошибки vector store;
- длительность запросов;
- выбранную модель;
- HTTP status;
- trace id/request id.

Frontend должен логировать в консоль только dev-информацию.

В production-like режиме frontend не должен засорять консоль лишними логами.

Нужно только не принимать архитектурных решений, которые помешают будущему переносу.

## 23. Критерии приёмки MVP

MVP считается готовым, если выполнены все условия ниже.

### 23.1. Запуск

- Проект запускается локально через Docker Compose.
- `.env.example` содержит все необходимые переменные.
- README объясняет запуск понятными командами.
- Backend открывается на локальном адресе.
- Frontend доступен через backend static serving.

### 23.2. Чат

- Пользователь может отправить текстовое сообщение.
- Ответ приходит от VseLLM.
- Ответ отображается потоково.
- Выбранная модель реально используется в запросе.
- Системный промт реально используется в запросе.
- Можно очистить текущую сессию.
- После очистки контекст не используется.

### 23.3. Модели

- Список моделей подтягивается через backend.
- Пользователь может выбрать модель.
- Если у модели есть только ID, UI показывает ID.
- Если модель не найдена, пользователь получает понятную ошибку.

### 23.4. Настройки

- Можно изменить имя ассистента.
- Можно изменить системный промт.
- Можно выбрать модель.
- API-ключ не отображается.
- Статус API-ключа отображается.

### 23.5. Файлы

- Можно загрузить PDF/DOCX/XLSX/изображение.
- Файлы больше 256 МБ отклоняются.
- Более 10 файлов за сообщение отклоняются.
- Текстовые файлы индексируются через embeddings.
- Вопрос по документу использует релевантные chunks.
- Очистка сессии удаляет файлы и временный индекс.

### 23.6. Состояние

- `/api/health` работает.
- Страница «Состояние Asya» отображает ключевые статусы.
- При отсутствии API-ключа показывается понятная ошибка.

### 23.7. Безопасность

- API-ключ хранится только в backend `.env`.
- API-ключ не попадает в frontend.
- API-ключ не попадает в логи.
- `.env` находится в `.gitignore`.

### 23.8. Тесты

- Есть backend tests для критичных endpoints.
- Есть frontend tests для основных экранов.
- Команда запуска тестов описана в документации.

## 26. Итоговая формулировка задания для разработчика

Необходимо разработать MVP персонального ИИ-ассистента Asya в виде PWA-веб-чата с локальным backend в Docker. Backend должен интегрироваться с VseLLM через OpenAI-compatible API, поддерживать выбор модели, потоковую генерацию, редактируемый системный промт, временные backend-сессии и работу с файлами через embeddings + локальную временную векторную базу.

Проект должен быть удобен для дальнейшей разработки через Codex / Claude Code: понятная структура, TypeScript на frontend, FastAPI/OpenAPI на backend, Docker Compose, тесты и актуальная документация.
