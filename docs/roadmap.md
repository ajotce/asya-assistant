## Принципы плана

**Принцип 1: каждая версия — рабочий продукт, не «стройка».** После каждого минорного релиза Asya должна быть пригодна к ежедневному использованию. Версия не считается завершённой, пока не работает целиком.

**Принцип 2: фичи добавляются группами, не россыпью.** Родственные фичи делаются в одной фазе — общий код, общие миграции, общие тесты.

**Принцип 3: инфраструктурное ядро — приоритет над фичами.** Multi-user, БД, авторизация, шифрование, фоновые задачи делаются раньше памяти, наблюдателя и интеграций.

**Принцип 4: голос — финальная фаза.** Real-time STT/TTS, wake-word, barge-in — самая сложная техническая часть. На v1.0 хватит push-to-talk + Telegram голосовых.

**Принцип 5: безопасность с первого дня.** Шифрование секретов в БД, master-key в `.env`, никаких токенов в логах — закладывается в 0.2 и не пересматривается.

## Версия 0.2 — Multi-user foundation

**Цель версии:** превратить Asya из одноюзерного локального приложения в multi-user систему с БД, миграциями, авторизацией и шифрованием — но без открытой регистрации. Пользователи создаются вручную через CLI.

**Что входит:** K1, K2, K3, A1, A2, A3, A4, B1 (перенос в БД), D1, D2.

**Что не входит:** никакой памяти кроме контекста сессии, никакого голоса, никаких интеграций, никакого наблюдателя, никаких пользовательских пространств, никакой открытой регистрации, никакого magic-link.

**Готовность к чему:** Asya перестаёт быть «локальной для одного». Можно создать 2–3 учётки для близких людей через CLI и собирать первую обратную связь. Фичей у Asya не больше, чем сейчас, но архитектура multi-user полностью заложена.

**Главный риск:** скучно делать. Нет «вау». Самое важное — не сорваться в параллельные фичи. Ничего, кроме фундамента, в 0.2 не делается.

**Ветка:** `0.2-multi-user`.

## Версия 0.3 — Память и личность

**Цель версии:** дать Asya настоящую память, чтобы она помнила пользователя между разговорами, узнавала стиль общения, могла ссылаться на прошлые разговоры. Плюс — возможность создавать собственные пространства (workspaces).

**Что входит:** B2, B3, B4, B5 (базовый), B6, B7, B8, B9, B11, B12, B13, B14, D3, D4, J3.

**Что не входит:** дневник (0.4), B10 (наблюдатель), B5 в полной форме с эволюцией (0.5), D5 (приватный чат — 0.4), интеграции, голос.

**Готовность к чему:** Asya становится «помнящей». Узнаёт стиль общения, помнит факты, ссылается на прошлые разговоры, в каждом пространстве может вести себя по-разному. Интеграций ещё нет — работает только с тем, что пользователь сам рассказывает.

**Главный риск:** загрязнение памяти. Особенно слой B5 (личность ассистента) — если он начнёт «учиться» с самого начала на сырых разговорах, может уехать в нежелательную сторону. Обязательно: версионирование (B7), снэпшоты (B8), возможность отката (B6), явный режим «наблюдение» перед «записью».

**Перед началом:** провести исследование R4 (VseLLM возможности) — нужно знать, какие модели поддерживают эмбеддинги, и какой стек выбрать.

#### Задача 0.4.13 — Тихий наблюдатель: правила-детекторы

Базовый набор детекторов на 0.4:

- `UnansweredImportantEmailDetector` — письмо от важного отправителя без ответа > 3 дней. «Важный» определяется по эвристике: письма, на которые пользователь обычно отвечает в течение суток.
- `MeetingConflictDetector` — пересечение встреч в календаре.
- `BackToBackMeetingsDetector` — три встречи подряд без перерыва.
- `BurnoutPatternDetector` (B10) — 5+ часов встреч в день несколько дней подряд + поздние сообщения. Включён по умолчанию, можно отключить в настройках.

Каждый детектор реализуется как отдельный класс, наследуется от `BaseDetector` с методом `detect(user_id) -> list[Observation]`.

**Acceptance:** все 6 детекторов реализованы, тесты с моками работают, в реальной интеграции при наличии триггерных данных создаются наблюдения.

#### Задача 0.4.19 — Приватный чат с шифрованием от админа (D5)

- Новое пространство типа `private_encrypted`.
- В этом пространстве каждое сообщение шифруется отдельным ключом, который выводится из master-key + пароля пользователя (KDF, например Argon2).
- Админ, имеющий доступ к master-key, **не** может расшифровать содержимое без пароля пользователя.
- Сообщения в БД хранятся как `EncryptedMessage` (отдельная таблица).
- При отображении в UI — Asya расшифровывает на лету, используя пароль из текущей сессии.
- Если пользователь меняет пароль — старые приватные сообщения становятся недоступными (или пользователь должен ввести старый пароль для перешифровки — решение в `docs/decisions.md`).
- Память (профиль, эпизоды, правила) от приватного чата **не пишется**: `write_to_memory=False` принудительно.

**Acceptance:** в приватном чате сообщения сохраняются зашифрованными в БД; ручная проверка через прямой запрос к БД показывает только blob; пользователь может вернуться через час, ввести пароль, увидеть историю; админ через `sqlite3` или Postgres-клиент не может прочитать содержимое.

Документы автоматизированы (создаются по голосовой команде, заполняются с фото VIN/паспорта), все источники подключены, утренний и вечерний брифинги работают, наблюдатель отслеживает паттерны во времени. Похоже на v1.0, но без полноценного голоса (только push-to-talk) и без Apple.

**Главный риск:** объём задач в 0.5 большой и разнородный (6 новых интеграций + документы + брифинги + эволюция личности). Велик риск растягивания. **Совет:** делать строго по разбивке ниже, не пытаться параллелить. Вторая опасность — H3 (распознавание VIN с фото): качество vision-моделей нестабильно, ошибка в VIN на гарантии — серьёзная проблема. Закладывать обязательную валидацию (длина 17 символов, контрольная цифра) и подтверждение пользователем перед вставкой в документ.

**Перед началом:** R2 (Bitrix24 audit) — обязательно, чтобы понимать какие методы API доступны в твоей редакции. R5 (маршрутизатор моделей) — опционально, если становится тесно с одной моделью.

**Ветка:** `0.5-extended`.

#### Задача 0.5.10 — Шаблоны документов: заполнение текстом (H2)

- Сервис `DocumentFillService`:
  - `fill_template(template_id, values: dict) -> filled_docx_bytes` — скачивает шаблон из Drive, подставляет значения вместо placeholder'ов, возвращает заполненный docx.
  - Использует `python-docx` для работы с docx.
- Голосовая команда / текст: «Ася, сделай гарантию для Иванова Ивана, VIN такой-то, машина Geely Monjaro, телефон такой-то, дата сегодня».
- Asya:
  - Распознаёт намерение (по NLU через основную LLM): «заполни шаблон 'гарантия' такими-то полями».
  - Если каких-то полей не хватает — спрашивает.
  - Заполняет шаблон через `DocumentFillService`.
  - Возвращает в чат файл (по решению пользователя — только в чат, не сохраняет в Drive).
- API: `POST /document-templates/{id}/fill` с body содержащим значения полей.

**Acceptance:** пользователь голосом просит заполнить шаблон, получает готовый docx-файл в чате.

#### Задача 0.5.11 — Шаблоны документов: распознавание VIN/паспорта с фото (H3)

- Зависит от того, поддерживает ли VseLLM Vision-модели (R4).
- Сервис `VisionService`:
  - `extract_vin(image_bytes) -> VinExtractionResult` — возвращает кандидаты VIN с confidence.
  - `extract_passport_number(image_bytes) -> PassportNumberResult`.
- Валидация:
  - VIN: длина 17 символов, корректные символы (без I, O, Q), валидная контрольная цифра.
  - Паспортный номер: формат страны (для РФ — 4 цифры серии + 6 цифр номера).
- Если confidence низкая или валидация не прошла — Asya спрашивает «Не уверена в распознанном VIN, проверьте: <значение>», даёт пользователю поправить.
- В чате при заполнении шаблона: пользователь может приложить фото VIN, Asya распознаёт и подставляет.
- **Запрет на тихую вставку нераспознанных значений**: если confidence < порога (например, 0.85) — обязательное явное подтверждение пользователем.

**Acceptance:** пользователь в чате прикрепляет фото VIN, Asya распознаёт и подставляет в нужное поле; при низком качестве фото — спрашивает подтверждение.

**Acceptance:** пользователь после заполнения шаблона получает оба файла; pdf корректно отображается, docx редактируется.

#### Задача 0.5.13 — Утренние и вечерние брифинги (I2, I3)

- Модель `Briefing` с полями: `id`, `user_id`, `kind` (`morning` / `evening`), `created_at`, `content` (markdown), `delivered_via` (in-app / telegram / both).
- Сервис `BriefingService.generate(user_id, kind)`:
  - Собирает данные: события из календаря, задачи (просрочки, сегодняшние), важные письма, наблюдения за день, опционально — записи дневника (если включено в C4).
  - Передаёт в LLM с промтом «составь утренний/вечерний брифинг», получает текст.
  - Сохраняет в `Briefing`.
- Расписание через scheduler из 0.4.3:
  - Утром в 8:00 (или по настройке пользователя) — генерация утреннего брифинга.
  - Вечером в 19:00 — генерация вечернего.
- Доставка:
  - In-app: уведомление в центре уведомлений с пометкой «Ваш утренний брифинг готов», клик — открывает страницу брифинга.
  - Telegram: автоматическая отправка в личку бота, если пользователь привязал Telegram и в настройках включена доставка туда.
- Настройки в личном кабинете: время утром/вечером, включить/отключить, каналы.
- Frontend: страница «Брифинги» с архивом за последние 30 дней.

**Acceptance:** утром в указанное время приходит брифинг с релевантным содержимым; в архиве видны прошлые брифинги.

#### Задача 0.5.14 — Откат действий (J4)

- Модель `ActionEvent` с полями: `id`, `user_id`, `action_kind`, `target_provider` (linear/calendar/...), `target_id`, `payload_before`, `payload_after`, `is_reversible`, `reversed`, `created_at`.
- Сервис `ActionEventService` логирует каждое действие Asya, выполняемое через интеграцию.
- Откатываемые действия:
  - Создание задачи в Linear/Todoist → удаление созданной.
  - Изменение статуса/приоритета → возврат к предыдущему значению.
  - Создание комментария → удаление.
  - Создание файла в Drive → удаление.
- Неоткатываемые действия:
  - Отправка email — после отправки undo невозможен.
  - Удаление в стороннем сервисе (если Asya удалила что-то — обратно не вернуть).
- Эндпоинт: `POST /actions/{id}/reverse` — попытка отката. Если `is_reversible=False` — 400 с пояснением.
- Frontend: в Ленте активности у каждой записи — кнопка «Откатить» (только для откатываемых).

**Acceptance:** пользователь создаёт задачу через Asya, откатывает через кнопку, задача удаляется в Linear; для отправленного email кнопка отката отсутствует.

#### Задача 0.5.15 — Снимки состояний для наблюдателя (E5)

- Модель `EntitySnapshot` с полями: `id`, `user_id`, `provider`, `entity_id`, `entity_type` (`task` / `event` / `email`), `snapshot_data` (JSON), `taken_at`.
- При каждом polling-цикле наблюдателя — для каждой отслеживаемой сущности создаётся snapshot текущего состояния.
- Старые snapshots постепенно удаляются: храним последние 30 дней, потом архивируем (с пониженной детализацией: вместо ежечасных — ежедневные, потом еженедельные).
- Расширение детекторов: теперь `RepeatedReschedulingDetector` имеет доступ к истории и может реально отследить «задача переносилась 4 раза за неделю».
- Дополнительные детекторы на основе истории:
  - `LongStaleTaskDetector` — задача без обновлений >30 дней.
  - `ProcrastinationPatternDetector` — конкретный тип задач постоянно переносится.
- Управление местом: настройка «глубина истории» (по умолчанию 30 дней, можно менять).

**Acceptance:** через 1–2 недели использования наблюдатель показывает паттерны, которые требуют истории (переносы, застревания).

#### Задача 0.5.16 — Эволюция личности ассистента (B5 в полной форме)

- Расширение базового B5 из 0.3.
- Сервис `PersonalityEvolutionService.evolve(user_id)`:
  - Анализирует последние N эпизодов, активные правила, профиль.
  - Через LLM генерирует обновлённый prompt личности.
  - Сравнивает с текущей версией: если изменения значительные — создаёт **диф-уведомление** для пользователя с превью «вот что я хочу обновить в своей личности», ждёт подтверждения.
  - Если пользователь подтверждает — создаётся новая версия. Если отклоняет — обновление не происходит, в audit log запись «отклонено».
  - Snapshot создаётся автоматически перед обновлением.
- Триггеры эволюции:
  - Раз в N взаимодействий (N=200 по дефолту, настраивается).
  - При накоплении большого числа новых правил (>10) — внеплановая эволюция.
  - Ручной запуск через UI.
- Защита от загрязнения: если эволюция предлагает «жёсткие» изменения (например, сильный сдвиг тона) — повышенный confidence-threshold, более явное подтверждение.

**Acceptance:** после длительного использования (>200 взаимодействий) Asya предлагает обновить свою личность; пользователь видит диф, подтверждает или отклоняет; откат на любую предыдущую версию работает.

#### Задача 0.5.17 — Влияние дневника на личность (C6)

- Дневник — отдельная сущность, не попадает в обычную память автоматически (зафиксировано в 0.4).
- Но: при эволюции личности (0.5.16) — обобщённые сигналы из дневника учитываются: общие темы, общий тон, общие переживания. Не конкретные записи и не цитаты.
- Реализация: `PersonalityEvolutionService` при сборе данных читает агрегаты из дневника (без выгрузки конкретных записей в LLM-промт):
  - Топ-10 тем за последние 30 дней.
  - Общий эмоциональный сдвиг (улучшение/ухудшение настроения по тренду).
  - Часто упоминаемые люди (без конкретного контекста).
- В диф-превью (0.5.16) — пользователь видит, что часть сигналов пришла из дневника, может явно отказаться от их учёта.

**Acceptance:** пользователь видит, что дневник повлиял на эволюцию личности, может отказаться (снимает галочку «учитывать дневник в эволюции»), и в следующий цикл сигналы дневника не используются.

## Версия 1.0 — Голос и публичный релиз

**Цель версии:** добавить полноценный голосовой режим (push-to-talk → wake-word), открыть регистрацию для широкой аудитории (OAuth + бета-программа), переехать в облако, поднять мониторинг и резервное копирование. Первая версия, которую можно открыто рекомендовать другим людям.

**Что входит:** G3 (wake-word в активной вкладке), G4 (режим «Слушаю»), A6 (OAuth), K6 (резервное копирование), K8 (JSON-выгрузка пользователя), переезд в облако с auto-scaling, миграция БД на PostgreSQL, S3-совместимое хранилище для файлов, мониторинг (Prometheus+Grafana или Sentry), CI/CD pipeline.

**Что не входит:** G5 (полноценный диалог с barge-in и interrupt) — это 2.0+. Apple-интеграции — это 2.0+. E2E-шифрование — это 2.0+ или never (зависит от R7).

**Готовность к чему:** v1.0 — это **публичный релиз**. После этой версии:
- Можно сделать сайт-визитку и открыть приём заявок в бету.
- Можно постепенно расширять круг пользователей.
- Можно анонсировать в своих каналах.
- Можно (опционально) начинать монетизацию.

**Главные риски:**

- Real-time голос с правильным wake-up и хорошим качеством — самая сложная техническая часть всего проекта. Может оказаться, что G3 в полном объёме не успевается, и v1.0 выходит с урезанным голосом (только push-to-talk + Telegram). Это нормально — голос вынесется в v1.5.
- Миграция SQLite → PostgreSQL может потребовать переписывания части кода (особенно если в 0.3 был использован `sqlite-vec`, у которого нет прямого аналога в Postgres — будет переход на pgvector).
- Переезд в облако — это смена стека. Если код не подготовлен к cloud-deploy (зависимости от локального диска, локальные пути, отсутствие env-конфигов) — миграция превращается в большой рефакторинг. **Совет:** в 0.5 заложить cloud-readiness заранее (см. задачу 0.5.18 — нужно отдельно проверить готовность).

**Перед началом:**
- **R1** (голос real-time) — обязательно, без него выбор стека невозможен. Сделать за 1–2 месяца до начала фазы.
- **R6** (БД и масштабирование) — обязательно, для решения по миграции.
- Финансовая оценка облака: бюджет на месячные расходы хостинга при N пользователях. Предварительно — простая модель «X пользователей × Y запросов × Z цена-за-запрос».

**Ветка:** `1.0-public`.

> **Замечание о детализации:** ниже разбивка на задачи на укрупнённом уровне. Точная разбивка зависит от результатов R1 и R6, которые ещё не проведены. Финальная детализация — при подходе к фазе.

#### Задача 1.0.10 — Wake-word (G3) и режим «Слушаю» (G4)

Зависит от результатов R1.

- Подключение Picovoice Porcupine (Free для разработки, Enterprise для продакшена) — или альтернативного wake-word провайдера.

- Frontend: разрешение микрофона, постоянное прослушивание в активной вкладке, индикация состояния («слушаю»/«говорит»/«молчу»).

**Acceptance:** в открытой вкладке Asya реагирует на wake-word, переходит в режим «Слушаю», обрабатывает команду, отвечает голосом.

## Версия 2.0+ — Apple и продвинутые фичи

### Подверсия 2.1 — Полноценный голосовой диалог (G5)

**Ключевая зависимость:** опыт эксплуатации G3, G4 из 1.0; возможно — повторное R1 с акцентом на barge-in и interrupt.

**Высокоуровневые задачи:**

- Реализация barge-in: пользователь может прервать Asya во время её ответа.
- Реализация interrupt: Asya перестаёт говорить, начинает слушать.
- Streaming STT: распознавание начинается до того, как пользователь договорил.
- Streaming TTS: ответ начинает воспроизводиться до того, как полностью сгенерирован LLM-ом.
- Обработка пересечения речи: правильное определение, кто говорит (пользователь vs Asya).
- Снижение задержки до приемлемой для естественного диалога (целевой показатель: <500ms между концом фразы пользователя и началом ответа Asya).

**Главный риск:** настоящий barge-in — это передний край исследований real-time AI. Может оказаться, что для качественной реализации придётся менять backend-стек или использовать специализированных провайдеров (Pipecat, LiveKit, Daily.co и подобные).

**Acceptance:** разговор с Asya по голосу ощущается как разговор с человеком — можно перебивать, переспрашивать, прерывать.

### Подверсия 2.2 — Общие пространства (D6)

**Ключевая зависимость:** обратная связь от реальных пользователей 1.0 — нужны ли вообще общие пространства, и в каком виде.

**Высокоуровневые задачи:**

- Расширение модели `Space`: добавить поля `is_shared`, `members` (M2M к User), `roles` (owner / editor / viewer).
- Логика памяти в shared-spaces: общие эпизоды, общие правила. Защита от конфликтов: если два пользователя добавляют противоречащие правила — Asya замечает и предлагает разобраться.
- Приглашение в shared-space: через invite-токен или email (если email-flow к этому моменту реализован).
- Видимость: кто что добавил, аудит-лог per-space.
- Уведомления: при значимых изменениях в shared-space — уведомление другим участникам.

**Главный риск:** конфликты между пользователями могут портить общую память. Нужны механизмы изоляции и согласования.

**Acceptance:** семья или маленькая команда может вести общее пространство «Семья» / «Команда» с общими задачами, событиями и правилами.

### Подверсия 2.3 — E2E-шифрование (K7)

**Ключевая зависимость:** R7 — обязательно. Возможно, после R7 решение будет «не делаем», потому что zero-knowledge сильно ограничивает функциональность AI-ассистента.

**Возможные сценарии после R7:**

- **(a) Полный E2E** — все данные пользователя зашифрованы клиентским ключом, админ ничего не видит. Сильно ограничивает фичи (Asya не может анализировать данные, которые не может прочитать).
- **(b) Частичный E2E** — только определённые типы данных зашифрованы (приватный чат D5 — уже есть, дневник, медицинская/финансовая информация). Остальное — обычное шифрование с master-key.
- **(c) Не делать** — оставить как есть (шифрование секретов, шифрование приватного чата D5). E2E не вводить.

**Рекомендация (предварительная, до R7):** скорее всего (b) — частичный E2E. Делать E2E для всего бессмысленно, потому что половина ценности Asya — в анализе данных.

**Acceptance (если делается):** определённые типы данных зашифрованы клиентским ключом, админ через прямой запрос к БД видит только blob.

### Возможные дополнения в 2.0+ (не запланированы, кандидаты на обсуждение)

Это вещи, которые **могут** оказаться нужны после v1.0 в зависимости от обратной связи. Сейчас не планируются, но фиксируются как кандидаты:

- Маршрутизатор моделей (п.47) — если станет ясно, что одна модель не справляется с разными типами задач. Опирается на R5.
- Кастомные wake-words — пользователь может задать своё слово.
- Plugin-система: возможность для пользователей добавлять свои интеграции через готовое API.
- Командная подписка (если общие пространства окажутся востребованы): тарифные планы для команд.

## Архитектурные решения, общие для всех версий

### Стек

- Backend: Python 3.11+, FastAPI, SQLAlchemy 2.x, Alembic, Pydantic 2.x.
- Frontend: React + Vite + TypeScript, существующая архитектура.
- БД: SQLite на 0.2–0.5, PostgreSQL начиная с 1.0.
- Векторное хранилище: `sqlite-vec` на 0.3–0.5, pgvector с 1.0.
- Шифрование: `cryptography.fernet`, master-key в `.env`.
- LLM: VseLLM (OpenAI-compatible) — основной провайдер, OpenRouter как запасной.
- Эмбеддинги: на 0.3 — локальная sentence-transformers модель, потом ревизия.
- Фоновые задачи: APScheduler на 0.4, Celery+Redis рассмотреть к 1.0.
- Контейнеризация: Docker Compose до 1.0, k8s/Yandex Cloud Managed Services с 1.0.

### Безопасность

- Все секреты — в `.env` (на dev) или secret manager (на проде с 1.0).
- Все токены интеграций — в БД зашифрованные через master-key.
- Master-key — отдельная переменная окружения, никогда не в коде, никогда не в логах.
- В `.env.example` — только примеры без реальных значений.
- Логи никогда не содержат: пароли, master-key, токены интеграций, содержимое сообщений (только метаданные).
- При экспорте данных пользователя (K8) — токены интеграций исключаются.

### Что нельзя делать в коде

- Использовать локальные пути к файлам (всегда через env-vars или абстракцию хранилища).
- Хардкодить пути или ID — только через конфиг.
- Делать массовые удаления в API без флага явного подтверждения.
- Создавать аккаунты пользователей в сторонних сервисах от имени пользователя.
- Записывать пароли пользователей даже на короткое время (всегда сразу хеш).
- Логировать тело запросов к LLM в продакшене (только метаданные).
