# Asya - Multi-User Foundation (v0.2) Acceptance Criteria

## Overview
This document contains the acceptance criteria for version v0.2 of the Asya project, codenamed "Multi-user foundation". 

## Criteria List

### 1. Infrastructure and Dependencies
- [ ] A new branch `0.2-multi-user` is created from `main`.
- [ ] `docs/roadmap.md`, `docs/acceptance/v0.2.md` and `docs/acceptance/v0.3.md` are added.
- [ ] `AGENTS.md` and `CLAUDE.md` are updated to reflect phase 0.2.
- [ ] `README.md` is updated to mention the CLI registration in v0.2.
- [ ] Required dependencies are added: `sqlalchemy`, `alembic`, `cryptography`, `passlib[bcrypt]`, `python-jose[cryptography]` (or `itsdangerous`).
- [ ] Session choice (JWT vs cookies) is documented in `docs/decisions.md`.
- [ ] `.env.example` includes `ASYA_MASTER_KEY`, `ASYA_DATABASE_URL`, `ASYA_SESSION_SECRET`.
- [ ] `docker-compose.yml` has a volume setup for the SQLite file.

### 2. Database Models and Migrations
- [ ] `backend/db/` package is created with `base.py` and `session.py`.
- [ ] Models defined: `User`, `Session`, `Chat`, `Message`, `FileMeta`, `UsageRecord`, `EncryptedSecret`.
- [ ] All tables containing user data include `user_id` foreign keys.
- [ ] `User` model contains `password_hash`.
- [ ] The schema is documented in `docs/architecture.md`.
- [ ] Alembic is initialized and configured in `backend/db/migrations`.
- [ ] Initial migration `0001_initial.py` generates the required tables in SQLite.
- [ ] Alembic upgrade/downgrade commands work correctly and are documented in `docs/development.md`.

### 3. Encryption and Security
- [ ] `backend/security/encryption.py` is implemented with `encrypt` and `decrypt` via `cryptography.fernet`.
- [ ] App crashes on startup if `ASYA_MASTER_KEY` is not present.
- [ ] `EncryptedSecret`-helper utility is added.
- [ ] Master-key or user passwords are NOT exposed in logs.
- [ ] Tests for encryption module verify proper key handling and encryption integrity.

### 4. Storage Refactoring
- [ ] In-memory `SessionStore` is rewritten to use SQLAlchemy `Session` and `Message` models.
- [ ] `VectorStore` uses `sqlite-vec` (or fallback to BLOB if sqlite-vec is not ready).
- [ ] `UsageStore` utilizes a database table to save stats.
- [ ] Chat history, vectors, and usage stats persist across application restarts.
- [ ] Relevant unit tests are updated to use in-memory SQLite.

### 5. Authentication and Authorization
- [ ] Authentication endpoints `/auth/login`, `/auth/logout`, `/auth/me` are implemented.
- [ ] Middleware/dependencies enforce valid sessions for protected routes (returning 401 Unauthorized if missing).
- [ ] Passwords are hashed using `passlib[bcrypt]`.
- [ ] Users are strictly isolated: A user cannot access another user's chats, sessions, or files via ID.
- [ ] Unit tests for auth functionality (login, invalid password, session expiry, cross-user isolation) pass.

### 6. Command Line Interface (CLI)
- [ ] `backend/cli.py` is created with commands: `user create`, `user list`, `user reset-password`, `user delete`.
- [ ] CLI commands use the exact same DB models and encryption layer as the API.
- [ ] CLI functionality is documented in `README.md` and `docs/development.md`.

### 7. Chats and Spaces Management
- [ ] `Chat` model is implemented, and a `Base-chat` is automatically created for every new user.
- [ ] API endpoints for chats: `GET /chats`, `POST /chats`, `DELETE /chats/{id}`, `PATCH /chats/{id}`.
- [ ] `Base-chat` cannot be deleted.
- [ ] Frontend UI shows a list of chats, allows switching, renaming, and creating new ones.

### 8. User Settings
- [ ] The `/settings` page allows password changes and light/dark theme toggling.
- [ ] API endpoints: `POST /auth/change-password`, `GET /me/preferences`, `PATCH /me/preferences`.
- [ ] Settings (like theme) persist across sessions.

### 9. Documentation and Tests
- [ ] `pytest backend/tests` passes entirely.
- [ ] `docs/api.md` lists all new endpoints.
- [ ] `docs/development-log.md` is updated with v0.2 chronology.
- [ ] Alembic setup with an empty database, user creation via CLI, and subsequent login/usage work flawlessly.
- [ ] Database downgrade and upgrade work without issue.
