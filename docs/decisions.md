# Decisions (Asya 0.2 Foundation)

Дата обновления: 2026-05-02

## ADR-001: Локальный режим остаётся базовым режимом разработки
Решение: backend продолжает запускаться локально в Docker Compose, frontend работает same-origin через backend-раздачу.
Причина: это текущий стабильный режим, который упрощает переход на 0.2 без параллельной смены инфраструктуры.

## ADR-002: API-ключ и чувствительные секреты только на backend
Решение: `VSELLM_API_KEY` и master/encryption key хранятся только в backend `.env` и не попадают во frontend, логи и клиентский трафик.
Причина: базовое требование безопасности перед внедрением multi-user auth.

## ADR-003: Переход проекта к Asya 0.2 — Multi-user foundation
Решение: приоритет этапа — фундамент multi-user: SQLite + Alembic, SQLAlchemy-модели, users/auth, beta invite/access request, multiple chats, обязательный `Base-chat`, миграция runtime-данных в БД, привязка данных к `user_id`, шифрование секретов.
Причина: текущий runtime/in-memory контур достаточен для single-user, но не обеспечивает надёжность и изоляцию данных для multi-user.

## ADR-004: Границы версии 0.2
Решение: в рамках 0.2 не добавлять новые крупные функции вне фундамента (память, личность, голос, дневник, внешние интеграции, web search, мобильное приложение, сложный billing) без отдельного явного запроса.
Причина: контроль scope и снижение архитектурного риска во время миграции на users/auth/db.

## ADR-005: Разделение слоёв и обратная совместимость API в переходный период
Решение: изменения делать поэтапно и сохранять работоспособность текущих endpoint-ов (`/api/health`, `/api/models`, `/api/settings`, `/api/session*`, `/api/chat/stream`, `/api/usage*`) либо документировать миграцию.
Причина: минимизация регрессий и возможность безопасно перенести in-memory storage в persistent слой.

## ADR-006: Auth v1 как минимальный собственный слой (без FastAPI-Users)
Решение: для Asya 0.2 внедряется минимальный собственный auth-слой на текущем FastAPI-коде. Password hash реализован через `hashlib.pbkdf2_hmac` (без дополнительной production-зависимости), сессия — opaque-token в `HttpOnly` cookie, а в БД хранится только hash токена (`auth_sessions.session_token_hash`) с `expires_at` и `revoked_at`.
Причина:
- меньше интеграционного риска, чем ранняя интеграция FastAPI-Users в уже работающий код;
- прямой контроль над текущей схемой `users/auth_sessions` и политиками `pending/active/disabled`;
- проще поэтапно подключать существующие endpoint-ы к `current user` без рефакторинга всего API за один шаг.
