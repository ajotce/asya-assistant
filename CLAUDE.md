# CLAUDE.md

> Этот файл — тонкая обёртка для Claude Code. Все основные правила работы — в `AGENTS.md`. Здесь только Claude-специфика.

## Главное

Прочитай `AGENTS.md` перед началом работы. Он — единственный источник правды о проекте, текущей фазе (1.0 — Voice & Public Release), запретах и стиле работы.

После `AGENTS.md` прочитай `docs/roadmap.md` (раздел «Версия 1.0») и `docs/acceptance/v1.0.md` (критерии приёмки текущей фазы; создаётся в задаче 1.0.1).

Для пошаговых промтов мультиагентной разработки — см. `docs/v1.0-prompts.md`.

## Claude-специфика

### TodoWrite для отслеживания задач

Когда задача состоит из нескольких шагов (3+), используй TodoWrite, чтобы видеть прогресс. Помечай шаги как `in_progress` перед началом и `completed` сразу после завершения, не накапливай статусы. В фазе 1.0 типичная задача (например, миграция на PostgreSQL) — это 5–10 шагов, веди их явно.

### Plan mode для нетривиальных изменений

Перед любым изменением, которое затрагивает больше двух файлов или меняет архитектуру (типичные примеры в 1.0: PG-миграция, S3-абстракция, OAuth-провайдеры, wake-word), переходи в plan mode и согласовывай план с пользователем до начала кода.

### Extended thinking при сложных решениях

Когда задача связана с архитектурным выбором (например, как организовать миграцию SQLite → PostgreSQL без даунтайма, как структурировать S3-провайдеры, какой стек wake-word выбрать) — думай развёрнуто перед ответом, не торопись.

### Чтение перед редактированием

Перед редактированием файла обязательно читай его целиком (или релевантные части), даже если изменение кажется тривиальным. Маленькие правки часто ломают невидимый контекст.

### Не запускай разрушающие команды без подтверждения

`rm -rf`, `git push --force`, `DROP TABLE`, `alembic downgrade base` на боевой БД, `terraform destroy`, `kubectl delete`, удаление S3-bucket'ов — всегда сначала спрашивай. Даже если задача формально это подразумевает.

### Sub-agents для параллельных задач

В фазе 1.0 многие задачи выполняются параллельно (см. этапы в `docs/v1.0-prompts.md`). Если ты — основной агент в сессии, делегируй sub-agents'ам:

- **Explore**: «найди все места, где код пишет в локальный диск», «найди все hardcoded paths». Полезно для cloud-readiness audit (1.0.1).
- **general-purpose**: ресёрч (R1, R6 — голос, БД), независимый review архитектуры.

Не делегируй задачи, требующие глубокого понимания контекста проекта — их веди сам.

### Worktrees для параллельной работы

Если работаешь над несколькими задачами одновременно (например, K8 export и Postgres-миграция), используй git worktrees: каждая задача — в своём worktree, своя ветка `1.0/<task>-<name>`. Не миксуй изменения в одной рабочей копии.

### Префикс коммитов

В фазе 1.0 — обязательно префикс задачи: `[1.0.3] PostgreSQL: ...`, `[1.0.9] OAuth Yandex: ...`. Это помогает при ревью и при сборке релиз-нотов.
