# CLAUDE.md — правила работы Claude Code над Asya Assistant

Этот файл задаёт рабочие правила для Claude Code в репозитории `ajotce/asya-assistant`.

Главный файл проектных правил: `AGENTS.md`. Если есть сомнения или конфликт приоритетов, сначала следуй `AGENTS.md`, затем этому файлу, затем текущему сообщению пользователя.

## 1. Контекст проекта

Asya Assistant переходит от локальной одноюзерной версии **Asya Local** к версии **Asya 0.2 — Multi-user foundation**.

Цель 0.2: создать безопасный фундамент мультипользовательской версии:
- БД и миграции;
- пользователи;
- авторизация;
- beta invite / access request flow;
- множественные чаты;
- обязательный `Base-chat`;
- перенос runtime/in-memory данных в persistent storage;
- привязка данных к `user_id`;
- шифрование секретов;
- сохранение текущего чата, файлов, retrieval, settings, usage и streaming.

Не добавляй в 0.2 память, личность, голос, дневник, тихого наблюдателя, Telegram, Linear, Todoist, Gmail, GitHub-интеграции, web search и другие крупные функции без отдельного явного запроса.

## 2. Перед началом каждой задачи

1. Проверь, что открыт репозиторий `ajotce/asya-assistant`.
2. Проверь текущую ветку.
3. Для задач 0.2 используй ветку `0.2-multi-user`, если пользователь не указал иначе.
4. Прочитай `README.md`.
5. Прочитай `AGENTS.md`.
6. Прочитай этот `CLAUDE.md`.
7. Прочитай релевантные документы в `docs/`.
8. Посмотри текущую реализацию кода, который собираешься менять.
9. Дай короткий план и только потом вноси изменения.

## 3. Стиль работы

- Работай маленькими завершёнными шагами.
- Не расширяй scope Asya 0.2: любые задачи вне фундамента multi-user (db/auth/chats/security) выполняй только после явного подтверждения пользователя.
- Не переписывай архитектуру целиком.
- Не добавляй новые production-зависимости без объяснения.
- Сначала сохраняй стабильность текущего проекта, затем расширяй.
- Если задача большая, раздели её на несколько безопасных этапов.
- Если есть риск потери данных, утечки секретов или поломки auth, остановись и явно опиши риск.
- Не задавай много вопросов. Спрашивай только если продолжать действительно опасно или невозможно.

## 4. Технические ориентиры

Сохраняй текущую базу проекта:
- backend: FastAPI;
- frontend: React + Vite + TypeScript;
- запуск: Docker Compose;
- frontend раздаётся через backend same-origin;
- модели идут через VseLLM/OpenAI-compatible API;
- `VSELLM_API_KEY` хранится только на backend.

Для 0.2 допускаются изменения:
- добавление SQLAlchemy/Alembic;
- добавление auth-слоя;
- перенос session/file/usage storage в БД;
- расширение API для users/chats/auth/invites;
- базовый UI входа, личного кабинета и списка чатов.

## 5. Безопасность

- Никогда не коммить `.env`, API-ключи, токены, приватные данные и временные файлы.
- Не выводи секреты в логах и пользовательских ответах.
- Используй `.env.example` только с пустыми или демонстрационными значениями.
- Пароли хранить только в виде hash.
- Magic-link токены не логировать целиком.
- После auth обязательно проверять, что пользователь не может открыть чужие чаты, файлы и usage.

## 6. Документация и проверки

При изменении поведения обновляй документы:
- `README.md`;
- `docs/architecture.md`;
- `docs/api.md`;
- `docs/development.md`;
- `docs/testing.md`;
- `docs/development-log.md`;
- `docs/decisions.md`.

Перед завершением задачи запусти доступные проверки:

```bash
make test
make lint
make build-frontend
```

Если проверку нельзя запустить, напиши почему.

## 7. Формат финального ответа

Пиши по-русски, просто и понятно.

В конце задачи укажи:
1. что сделано;
2. какие файлы изменены;
3. какие проверки запущены;
4. что не проверено и почему;
5. какие риски или следующие шаги остались.

## 8. Главный принцип

Asya 0.2 должна стать надёжным фундаментом для будущей памяти, личности, интеграций и голосового режима. Не усложняй 0.2 будущими функциями раньше времени.
