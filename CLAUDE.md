# CLAUDE.md — правила работы Claude Code над Asya Assistant

Этот файл задаёт рабочие правила для Claude Code в репозитории `ajotce/asya-assistant`.

Главный файл проектных правил: `AGENTS.md`. Если есть сомнения или конфликт приоритетов, сначала следуй `AGENTS.md`, затем этому файлу, затем текущему сообщению пользователя.

## 1. Контекст проекта

Asya Assistant перешла к этапу **Asya 0.3 — Memory and Personality**.

Базовая ветка текущего этапа: `0.2-multi-user`.

Цель 0.3: добавить управляемую долговременную память, базовую личность ассистента, пользовательские пространства, ленту памяти и ленту активности без внешних интеграций и голосового режима.

В 0.3 входят:
- профиль пользователя с фактами и статусами;
- эпизодическая память с embeddings;
- правила поведения и предпочтения пользователя;
- базовый профиль личности ассистента;
- per-space настройки памяти и personality overlay;
- пользовательские пространства;
- служебное admin-only пространство `Asya-dev`;
- лента памяти;
- версионирование и снэпшоты памяти;
- лента активности;
- интеграция relevant memory/rules/personality в chat context.

Не добавляй в 0.3:
- Google/Apple/Gmail/Todoist/Linear/Bitrix24/GitHub/Telegram интеграции;
- голос, wake word, STT/TTS;
- дневник;
- тихого наблюдателя;
- центр уведомлений;
- шаблоны документов;
- приватный E2E-чат;
- web search;
- биллинг.

## 2. Перед началом каждой задачи

1. Проверь, что открыт репозиторий `ajotce/asya-assistant`.
2. Проверь текущую ветку.
3. Для задач 0.3 работай от ветки `0.2-multi-user`, если пользователь не указал иначе.
4. Проверь `git status`.
5. Прочитай `README.md`.
6. Прочитай `AGENTS.md`.
7. Прочитай этот `CLAUDE.md`.
8. Прочитай релевантные документы в `docs/`.
9. Посмотри текущую реализацию кода, который собираешься менять.
10. Дай короткий план и только потом вноси изменения.

## 3. Стиль работы

- Работай маленькими завершёнными шагами.
- Сначала изучай текущий код, потом меняй.
- Не расширяй scope 0.3.
- Не переписывай архитектуру целиком.
- Не добавляй production-зависимости без необходимости.
- Если задача большая, раздели её на несколько безопасных этапов.
- Если есть риск потери данных, утечки секретов, поломки auth или смешивания памяти пользователей — остановись и явно опиши риск.
- Не задавай много вопросов. Спрашивай только если продолжать действительно опасно или невозможно.

## 4. Технические ориентиры

Сохраняй текущую базу:
- backend: FastAPI;
- frontend: React + Vite + TypeScript;
- запуск: Docker Compose;
- frontend раздаётся через backend same-origin;
- модели и embeddings идут через VseLLM/OpenAI-compatible API;
- БД: SQLite + SQLAlchemy + Alembic;
- auth и user-scoped данные уже являются фундаментом 0.2;
- `VSELLM_API_KEY` хранится только на backend.

Для 0.3 допускаются изменения:
- SQLAlchemy-модели и Alembic миграции для memory/personality/spaces/activity;
- repository/service слой памяти;
- user-scoped API для памяти и пространств;
- расширение chat context за счёт релевантной памяти;
- UI для памяти, правил, активности и пространств;
- документация и тесты.

## 5. Память

Минимальная модель:
- факты профиля пользователя;
- эпизоды;
- embedding chunks;
- правила поведения;
- профиль личности ассистента;
- per-space настройки;
- версии изменений;
- снэпшоты;
- activity log.

Статусы: `confirmed`, `inferred`, `needs_review`, `outdated`, `forbidden`, `deleted`.

Правила:
- “запомни” -> можно сохранять как `confirmed`;
- автоматическое извлечение -> `inferred` или `needs_review`;
- “забудь” -> `forbidden` или `deleted`;
- `forbidden`/`deleted` память нельзя использовать в ответах.

## 6. Личность

v0.3 — только базовая управляемая личность:
- адаптация тона;
- лёгкий юмор;
- мягкое возражение;
- обращение по имени в важных моментах;
- извинение при ошибке;
- предложение сохранить правило после ошибки;
- base personality + space overlay.

Сложная автономная эволюция личности в 0.3 не входит.

Не имитировать сознание, эмоции или роль терапевта.

## 7. Безопасность

- Не коммить `.env`, токены, приватные данные и временные файлы.
- Не выводить секреты в логах и UI.
- Все memory/activity/space endpoint-ы должны проверять `user_id`.
- `Asya-dev` доступно только admin-пользователю.
- Не записывать в память пароли, токены, ключи и очевидно чувствительные данные автоматически.
- Не допускать cross-user data leakage.

Перед завершением задачи запусти:

```bash

make lint

```

Если проверку нельзя запустить, напиши почему.

## 9. Формат финального ответа

Пиши по-русски, просто и понятно.

В конце задачи укажи:
1. что сделано;
2. какие файлы изменены;
3. какие проверки запущены;
4. что не проверено и почему;
5. какие риски или следующие шаги остались.

## 10. Главный принцип

Asya 0.3 должна добавить память и личность без потери стабильности и безопасности multi-user фундамента 0.2.
